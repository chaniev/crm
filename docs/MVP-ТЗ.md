# MVP-ТЗ Gym CRM для спортивного зала

Дата: 2026-04-15

## 1. Назначение системы

Gym CRM предназначена для учета клиентов спортивного зала, групп, тренеров, абонементов, оплат и посещений.

Система должна позволять:

- вести базу клиентов;
- объединять клиентов в группы;
- назначать тренеров на группы;
- учитывать абонементы и оплаты;
- отмечать посещения тренировок;
- хранить данные выбранного абонемента клиента, историю изменений абонемента и историю посещений;
- вести журнал действий пользователей;
- разграничивать доступ по ролям.

## 2. Роли пользователей

В системе должны быть роли:

- Главный тренер;
- Администратор;
- Тренер.

### 2.2. Ограничения для тренера

Тренер должен видеть только клиентов тех групп, где он назначен тренером.

В MVP тренер видит по клиенту только:

- ФИО;
- фотографию;
- группы клиента, в которых этот тренер назначен тренером;
- элементы отметки посещения;
- предупреждение о проблеме с абонементом без раскрытия полных данных абонемента;
- иконку отсутствия оплаты, если абонемент не оплачен.

Тренер не должен видеть:

- телефон клиента;
- контактных лиц клиента;
- телефоны контактных лиц;
- полные данные абонемента;
- сумму оплаты и полные данные оплаты.

### 2.3. Пользовательский маршрут тренера

После входа в систему тренер проходит следующий маршрут:

- если для пользователя включен признак обязательной смены пароля, тренер сначала попадает на экран смены пароля;
- после успешного входа или смены пароля тренер попадает на экран отметки посещений;
- если у тренера нет назначенных групп, на экране отметки посещений должно отображаться пустое состояние с сообщением о том, что назначенные группы отсутствуют.

Тренеру доступны разделы:

- экран отметки посещений;
- список клиентов только по тем группам, где тренер назначен тренером, в режиме просмотра без редактирования;
- карточка клиента только для клиентов из назначенных тренеру групп;
- история посещений клиента только по назначенным тренеру группам;
- смена пароля;
- выход из системы.

Тренеру недоступны разделы и действия:

- главная страница с клиентами, у которых скоро заканчивается абонемент;
- управление пользователями;
- создание и редактирование групп;
- создание, редактирование, архивирование и возврат клиентов из архива;
- оформление, продление и исправление абонемента;
- отметка оплаты;
- журнал действий пользователей.

В списке клиентов и на экране отметки посещений тренер видит только:

- фотографию клиента;
- ФИО клиента;
- группы клиента, доступные этому тренеру;
- элементы отметки посещения для выбранной группы и даты;
- предупреждение о проблеме с абонементом без раскрытия полных данных;
- иконку отсутствия оплаты, если абонемент не оплачен.

В карточке клиента тренер видит только:

- фотографию клиента;
- ФИО клиента;
- группы клиента, доступные этому тренеру;
- историю посещений клиента по назначенным тренеру группам.

В истории посещений клиента тренер видит только:

- дату тренировки;
- группу, доступную этому тренеру;
- признак посещения.

## 3. Авторизация и пользователи

### 3.1. Авторизация

Пользователь входит в систему по логину и паролю.

При первом входе пользователь обязан сменить пароль.

### 3.2. Первый пользователь

При развертывании системы должен автоматически создаваться пользователь:

- роль: Главный тренер;
- пароль: `12345678`;
- признак обязательной смены пароля при первом входе: да.

Логин первого пользователя должен быть задан в конфигурации или создан по значению по умолчанию.

### 3.3. Карточка пользователя

Пользователь должен иметь поля:

- ФИО одним полем;
- логин;
- пароль;
- роль;
- признак обязательной смены пароля;
- статус активности.

После создания пользователя логин нельзя изменить.

Требования к сложности пароля в MVP не задаются.

Удаление пользователей в MVP не требуется. Достаточно статуса активности.

## 4. Клиенты

### 4.1. Карточка клиента

Для клиента должны храниться:

- фотография;
- фамилия;
- имя;
- отчество;
- номер телефона;
- контактные лица, не более 2;
- группы, в которые входит клиент;
- выбранный тип абонемента и параметры оплаты;
- история посещений;
- статус клиента.

При создании и редактировании клиента обязательны следующие поля:

- номер телефона;
- хотя бы одно из полей ФИО: фамилия, имя или отчество.

При создании клиента абонемент не обязателен.

Клиент может быть создан без текущего абонемента.

Абонемент может быть добавлен позже через карточку клиента.

### 4.2. Статус клиента

Клиент может иметь один из статусов:

- активный;
- архивный.

Пользователь с ролью Главный тренер или Администратор может:

- перевести клиента в архив;
- вернуть клиента из архива в активные.

Удаление клиентов в MVP не требуется.

### 4.3. Контактные лица клиента

Для контактного лица должны храниться:

- тип контактного лица;
- ФИО одним полем;
- номер телефона.

Тип контактного лица выбирается из вариантов:

- мама;
- папа;
- произвольный текст.

### 4.4. Фотографии клиентов

В MVP фотографии клиентов должны храниться как файлы на стороне backend.

В базе данных не должны храниться бинарные данные фотографии. В базе данных должны храниться только сведения о файле:

- путь или ключ файла;
- MIME-тип;
- размер файла;
- дата и время загрузки.

Файлы фотографий должны храниться в директории загрузок backend, подключенной как Docker volume, чтобы фотографии сохранялись между перезапусками контейнеров.

Допустимые форматы загрузки:

- JPEG: `.jpg`, `.jpeg`, `image/jpeg`;
- PNG: `.png`, `image/png`;
- WebP: `.webp`, `image/webp`;
- HEIC: `.heic`, `image/heic`;
- HEIF: `.heif`, `image/heif`.

Недопустимые форматы должны отклоняться backend-ом.

Максимальный размер загружаемого файла: 10 MB.

Backend должен проверять не только расширение файла, но и MIME-тип.

При загрузке файлов `HEIC` или `HEIF` backend должен автоматически конвертировать изображение в веб-совместимый формат для хранения и отображения. Рекомендуемый целевой формат хранения: `JPEG` или `WebP`.

При сохранении фотографии backend должен генерировать уникальное имя файла. Имя файла, переданное пользователем, не должно использоваться как имя файла на диске.

В MVP достаточно хранить одну актуальную фотографию клиента. При загрузке новой фотографии старая фотография клиента заменяется.

Рекомендуется автоматически уменьшать фотографию до максимального размера 1024 px по большей стороне.

Доступ к фотографии клиента должен проверяться на backend.

Главный тренер и Администратор могут просматривать фотографии всех клиентов.

Тренер видит фото клиента, если клиент состоит хотя бы в одной группе, где этот тренер назначен тренером.

## 5. Группы

### 5.1. Карточка группы

Для группы должны храниться:

- название группы;
- время начала тренировки;
- график тренировок;
- список клиентов;
- список тренеров.

### 5.2. Клиенты в группах

Один клиент может состоять в нескольких группах.

Одна группа может содержать нескольких клиентов.

### 5.3. Тренеры в группах

У группы может быть несколько тренеров.

Назначенный тренер получает право отмечать посещения только в этой группе.

Главный тренер имеет право отмечать посещения в любой группе.

## 6. Абонемент и оплата клиента

### 6.1. Типы абонементов

В MVP должны быть типы абонементов:

- разовое посещение;
- месячный абонемент;
- годовой абонемент.

### 6.2. Общие правила

В MVP абонемент не создается пользователем как отдельный объект через отдельный сценарий.

Тип абонемента и параметры оплаты указываются в карточке клиента.

Во внутренней модели данных абонемент клиента должен храниться в отдельной таблице `ClientMembership`.

Карточка клиента показывает текущую запись `ClientMembership`.

Для текущего абонемента клиента должны храниться:

- выбранный тип абонемента;
- дата покупки;
- дата окончания;
- сумма оплаты;
- признак оплаты;
- признак использования разового абонемента;
- пользователь, отметивший оплату;
- дата и время отметки оплаты;
- причина создания версии абонемента.

Абонемент действует на все группы клиента.

Способ оплаты в MVP не хранится.

Отдельный пользовательский сценарий создания абонемента в MVP не требуется.

История изменений абонемента должна сохраняться через версии записей `ClientMembership`.

Для `ClientMembership` должны храниться технические поля:

- `ValidFrom`: дата и время начала действия версии записи;
- `ValidTo`: дата и время окончания действия версии записи, для текущей версии пустое значение.

Для версии `ClientMembership` должна храниться причина создания версии:

- оформление нового абонемента;
- продление текущего абонемента;
- исправление данных абонемента;
- отметка оплаты;
- автоматическое списание разового посещения.

У одного клиента должна быть только одна текущая запись `ClientMembership`, где `ValidTo` не заполнено.

При изменении типа абонемента, даты покупки, даты окончания, суммы оплаты или признака оплаты текущая версия `ClientMembership` закрывается, и создается новая текущая версия.

Эти же изменения должны фиксироваться в журнале действий пользователей.

### 6.3. Покупка, продление и исправление абонемента

В карточке клиента должны быть доступны действия:

- оформить новый абонемент;
- продлить текущий абонемент;
- исправить данные текущего абонемента;
- отметить оплату.

Действия выполняются в карточке клиента. Отдельный экран или отдельный раздел создания абонементов в MVP не требуется.

При создании клиента оформление абонемента не является обязательным шагом.

При оформлении нового абонемента:

- пользователь выбирает тип абонемента;
- пользователь указывает дату покупки;
- пользователь указывает сумму оплаты;
- пользователь указывает признак оплаты;
- текущая версия `ClientMembership` закрывается;
- создается новая текущая версия `ClientMembership`;
- дата окончания рассчитывается от даты покупки;
- для разового посещения признак использования устанавливается в значение "не использован".

При продлении текущего абонемента:

- тип абонемента сохраняется из текущей версии;
- пользователь указывает дату продления;
- пользователь указывает сумму оплаты;
- пользователь указывает признак оплаты;
- текущая версия `ClientMembership` закрывается;
- создается новая текущая версия `ClientMembership`;
- для месячного и годового абонемента базово новая дата окончания рассчитывается от текущей даты окончания абонемента;
- если с момента истечения предыдущего абонемента клиент ни разу не посещал занятия в течение одного месяца, новая дата окончания рассчитывается от даты оплаты;
- если указанное исключение не выполняется, срок продления добавляется к текущей дате окончания, а не к дате продления;
- для разового посещения продление считается оформлением нового разового посещения, признак использования устанавливается в значение "не использован".

При исправлении данных текущего абонемента:

- пользователь может изменить тип абонемента, дату покупки, дату окончания, сумму или признак оплаты;
- текущая версия `ClientMembership` закрывается;
- создается новая текущая версия `ClientMembership`;
- причина создания версии указывается как исправление данных.

При отметке оплаты:

- текущая версия `ClientMembership` закрывается;
- создается новая текущая версия `ClientMembership` с обновленным признаком оплаты;
- сохраняется пользователь, отметивший оплату;
- сохраняется дата и время отметки оплаты.

### 6.4. Срок действия

При оформлении нового месячного абонемента дата окончания автоматически подставляется как дата покупки плюс 1 месяц.

При оформлении нового годового абонемента дата окончания автоматически подставляется как дата покупки плюс 1 год.

Администратор или Главный тренер могут вручную изменить дату окончания абонемента.

При продлении месячного или годового абонемента применяется следующее правило:

- если с даты окончания предыдущего абонемента прошло менее одного месяца или в течение этого месяца у клиента было хотя бы одно посещение, новая дата окончания рассчитывается от текущей даты окончания предыдущего абонемента;
- если с даты окончания предыдущего абонемента прошел один месяц и более, и за этот период у клиента не было ни одного посещения, новая дата окончания рассчитывается от даты оплаты.

Для разового абонемента дата окончания не является основным правилом. Основное правило разового абонемента определяется признаком использования.

При выборе или обновлении разового абонемента в карточке клиента признак использования должен устанавливаться в значение "не использован".

### 6.5. Использование абонементов

Месячный и годовой абонементы являются безлимитными в пределах срока действия.

Разовый абонемент автоматически списывается после одного отмеченного посещения.

Отметка посещения разрешена независимо от актуальности абонемента.

Если у клиента нет подходящего абонемента, абонемент не оплачен, истек или разовый абонемент уже использован, система должна показать предупреждение, но разрешить отметку посещения.

Если абонемент клиента не оплачен, на экране отметки посещений рядом с информацией о клиенте должна отображаться визуальная иконка отсутствия оплаты.

Иконка отсутствия оплаты не должна блокировать отметку посещения.

Рекомендуется, чтобы иконка имела текстовую подсказку или пояснение при наведении или нажатии.

## 7. Посещения

### 7.1. Основной сценарий отметки посещения

Пользователь открывает экран посещений.

Пользователь выбирает:

- группу;
- дату тренировки.

Система показывает список клиентов выбранной группы.

Напротив каждого клиента пользователь может поставить или изменить признак посещения.

### 7.2. Права на отметку посещений

Главный тренер может отмечать посещения в любой группе.

Тренер может отмечать посещения только в группах, где он назначен тренером.

Администратор не может отмечать посещения.

### 7.3. История посещений

Для каждой отметки посещения должны храниться:

- клиент;
- группа;
- дата тренировки;
- признак посещения;
- пользователь, который поставил или изменил отметку;
- дата и время действия.

Редактирование посещений задним числом разрешено:

- Главному тренеру в любой группе;
- Тренеру только в назначенных группах.

История посещений клиента должна быть доступна для просмотра согласно правам ролей.

## 8. Главная страница

После входа Главный тренер и Администратор должны попадать на главную страницу.

Тренер после входа должен попадать на экран отметки посещений.

Для Главного тренера и Администратора на главной странице должен отображаться список клиентов, у которых до истечения абонемента осталось менее 10 дней.

Для каждого клиента в списке рекомендуется показывать:

- ФИО клиента;
- тип абонемента;
- дату окончания абонемента;
- количество дней до окончания;
- признак оплаты.

## 9. Журнал действий пользователей

Система должна вести журнал действий пользователей.

Журнал доступен:

- Главному тренеру;
- Администратору.

Журнал недоступен Тренеру.

В журнале должны фиксироваться действия:

- вход пользователя в систему;
- смена пароля;
- создание клиента;
- редактирование клиента;
- перевод клиента в архив;
- возврат клиента из архива;
- создание группы;
- редактирование группы;
- назначение тренера в группу;
- создание пользователя;
- редактирование пользователя;
- оформление нового абонемента клиента;
- продление абонемента клиента;
- выбор или изменение типа абонемента клиента;
- изменение даты покупки, даты окончания или суммы оплаты;
- исправление данных абонемента клиента;
- отметка оплаты;
- отметка посещения;
- изменение отметки посещения.

Для записи журнала должны храниться:

- дата и время действия;
- пользователь;
- тип действия;
- тип объекта;
- идентификатор объекта;
- краткое описание действия;
- старое значение измененных полей;
- новое значение измененных полей.

История изменений должна храниться следующим образом:

- журнал действий является append-only: записи журнала не редактируются и не удаляются через пользовательский интерфейс;
- для действий создания старое значение не заполняется, новое значение содержит созданные данные;
- для действий редактирования старое значение содержит значения измененных полей до изменения, новое значение содержит значения после изменения;
- для действий перевода клиента в архив и возврата из архива должны храниться старый и новый статус клиента;
- для действий с абонементом история состояния дополнительно хранится через версии `ClientMembership`;
- для действий с абонементом в журнале должны храниться старые и новые значения измененных полей, а также причина изменения;
- для отметки посещения должны храниться старое и новое значение признака посещения, если отметка была изменена;
- пароль и хеш пароля не должны сохраняться в старых или новых значениях журнала, фиксируется только факт создания пользователя или смены пароля.

Старое и новое значение должно храниться в структурированном формате JSON.

На экране журнала действий должна быть возможность фильтрации:

- по периоду действия: дата с / дата по;
- по пользователю;
- по типу действия;
- по типу объекта.

### 9.1. Технические логи backend

Кроме журнала действий пользователей backend должен вести отдельные технические логи приложения.

Технические логи предназначены для диагностики:

- запуска приложения;
- ошибок приложения и интеграций;
- применения миграций;
- HTTP-запросов и ответов на уровне технического журнала.

Требования к хранению технических логов:

- технические логи должны сохраняться в файлы на стороне backend;
- файлы логов должны храниться в отдельной директории backend, подключенной как Docker volume, чтобы логи сохранялись между перезапусками контейнера;
- максимальный размер одного файла лога: `20 MB`;
- срок хранения файлов логов: `10` дней;
- имя файла лога должно содержать дату и время его создания;
- по истечении срока хранения старые файлы логов должны автоматически удаляться.

Требования к безопасности технических логов:

- технические логи не заменяют журнал действий пользователей и не используются вместо `AuditLog`;
- в технические логи нельзя записывать пароль, хеш пароля, auth-cookie, `CSRF`-токены и лишние персональные данные клиентов.

## 10. Экраны MVP

В MVP должны быть реализованы экраны:

- авторизация;
- смена пароля при первом входе;
- главная страница;
- список клиентов;
- создание клиента;
- редактирование клиента;
- карточка клиента с текущими данными абонемента и историей посещений;
- список групп;
- создание группы;
- редактирование группы;
- экран отметки посещений;
- список пользователей;
- создание пользователя;
- редактирование пользователя;
- журнал действий.

### 10.1. Поведение CRUD-интерфейса

После успешного выполнения операций создания, редактирования, архивирования, возврата из архива, изменения абонемента, отметки оплаты и изменения посещения интерфейс должен обновляться автоматически без обязательного нажатия отдельной кнопки "Обновить".

На текущем экране пользователь должен сразу видеть актуальное состояние данных:

- в списках клиентов, групп, пользователей и журнале действий;
- в карточке клиента;
- на экране отметки посещений;
- на главной странице.

Допускается наличие кнопки ручного обновления как вспомогательного действия, но она не должна быть основным способом получения актуальных данных после CRUD-операций.

## 11. Поиск и фильтрация

В MVP рекомендуется реализовать:

- поиск клиента по ФИО;
- поиск клиента по телефону для Главного тренера и Администратора;
- фильтр клиентов по группе;
- фильтр клиентов по статусу: активные или архивные;
- фильтр клиентов по признаку оплаты;
- фильтр клиентов по сроку окончания абонемента.

Дополнительно в MVP должна быть реализована фильтрация клиентов:

- без фотографии;
- без группы;
- без актуального оплаченного абонемента.

Под клиентом без актуального оплаченного абонемента в рамках фильтра понимается клиент, у которого отсутствует текущий оплаченный абонемент, либо срок действия текущего оплаченного абонемента истек, либо разовый абонемент уже использован.

## 12. Нефункциональные требования

### 12.1. Адаптивность

Интерфейс должен автоматически адаптироваться к мобильным устройствам.

Экран отметки посещений должен быть удобен для использования с телефона.

Все основные экраны MVP должны корректно работать на desktop, планшетах и мобильных телефонах:

- авторизация;
- смена пароля;
- главная страница;
- список клиентов;
- карточка клиента;
- создание и редактирование клиента;
- список групп;
- создание и редактирование группы;
- экран отметки посещений;
- список пользователей;
- создание и редактирование пользователя;
- журнал действий.

Минимальные требования к мобильной адаптации:

- интерфейс не должен требовать горизонтальной прокрутки страницы;
- таблицы и длинные списки должны иметь мобильное представление или корректную внутреннюю прокрутку;
- основные действия должны быть доступны без наведения мыши;
- интерактивные элементы должны быть удобны для нажатия пальцем;
- формы должны выстраиваться в одну колонку на узких экранах;
- текст, кнопки и поля ввода не должны перекрываться;
- экран отметки посещений должен позволять тренеру быстро выбрать группу, дату и поставить отметки посещения одной рукой.

Адаптивность должна проверяться минимум на ширинах:

- 390 px;
- 768 px;
- 1440 px.

### 12.2. Производительность

Система должна быть рассчитана на:

- до 10 000 клиентов;
- до 100 пользователей.

Списки клиентов, групп и посещений должны поддерживать пагинацию или иной механизм, не требующий загрузки всех записей сразу.

### 12.3. Безопасность

Пароли должны храниться в виде криптографического хеша.

Для авторизации браузерного frontend должна использоваться cookie-based авторизация.

JWT для авторизации браузерного frontend в MVP не используется.

Авторизационная cookie должна быть `HttpOnly`.

В production-режиме авторизационная cookie должна быть `Secure`.

Для изменяющих запросов должна быть реализована CSRF-защита.

Доступ к данным должен проверяться на уровне backend.

Frontend не должен быть единственным местом проверки прав.

Тренер не должен получать через API персональные данные клиентов, к которым у него нет доступа.

### 12.4. Аудит

Ключевые действия пользователей должны попадать в журнал действий.

Журнал действий не должен редактироваться через пользовательский интерфейс.

## 13. Технические требования

Предполагаемый стек:

- backend: C#, .NET 10;
- frontend: React;
- база данных: PostgreSQL;
- контейнеризация: Docker.

Каждый создаваемый сервис должен запускаться в Docker.

Для локального запуска рекомендуется предусмотреть `docker-compose` с сервисами:

- backend;
- frontend;
- PostgreSQL.

## 14. Данные и связи

Основные связи:

- клиент может состоять в нескольких группах;
- группа может содержать нескольких клиентов;
- группа может иметь нескольких тренеров;
- тренер может быть назначен в несколько групп;
- клиент может быть создан без текущей записи `ClientMembership`; при наличии абонемента у клиента есть одна текущая запись `ClientMembership` и история изменений абонемента;
- клиент может иметь несколько посещений в истории;
- действие в журнале связано с пользователем.

## 15. Критерии приемки MVP

MVP считается готовым, если:

- пользователь может войти в систему по логину и паролю;
- при первом входе система требует сменить пароль;
- при первом запуске создан Главный тренер с паролем `12345678`;
- Главный тренер может создавать и редактировать пользователей;
- Главный тренер и Администратор могут создавать и редактировать клиентов;
- Главный тренер и Администратор могут переводить клиентов в архив и возвращать их в активные;
- Главный тренер и Администратор могут создавать и редактировать группы;
- в группе можно назначить несколько тренеров;
- клиент может быть добавлен в несколько групп;
- Главный тренер может отмечать посещения в любой группе;
- Тренер может отмечать посещения только в назначенных группах;
- Тренер после входа попадает на экран отметки посещений;
- Тренер в списке клиентов своих групп видит фотографию, ФИО, свои группы клиента, элементы отметки посещения, предупреждение о проблеме с абонементом и иконку отсутствия оплаты;
- Тренер в карточке клиента видит фотографию, ФИО, свои группы клиента и историю посещений только по назначенным группам;
- Администратор не может отмечать посещения;
- при неоплаченном абонементе рядом с клиентом на экране посещений отображается иконка отсутствия оплаты;
- в карточке клиента можно выбрать разовое посещение, месячный или годовой абонемент;
- при создании и редактировании клиента должны быть заполнены номер телефона и хотя бы одно из полей ФИО;
- клиент может быть создан без абонемента;
- абонемент можно добавить позже через карточку клиента;
- в карточке клиента можно оформить новый абонемент;
- в карточке клиента можно продлить текущий абонемент;
- в карточке клиента можно исправить данные текущего абонемента;
- при оформлении нового месячного абонемента дата окончания рассчитывается от даты покупки плюс 1 месяц;
- при оформлении нового годового абонемента дата окончания рассчитывается от даты покупки плюс 1 год;
- при продлении месячного или годового абонемента новая дата окончания обычно рассчитывается от текущей даты окончания плюс срок абонемента;
- если с момента истечения предыдущего абонемента клиент ни разу не посещал занятия в течение одного месяца, новая дата окончания рассчитывается от даты оплаты;
- разовый абонемент автоматически списывается после одного посещения;
- при проблеме с абонементом система показывает предупреждение, но разрешает отметить посещение;
- сохраняется история посещений клиента;
- изменения данных абонемента и оплаты клиента сохраняются через версии `ClientMembership`;
- изменения данных абонемента и оплаты клиента записываются в журнал действий;
- на главной странице видны клиенты с абонементами, истекающими менее чем через 10 дней;
- действия пользователей записываются в журнал;
- Главный тренер и Администратор могут просматривать журнал действий;
- журнал действий поддерживает фильтрацию по периоду, пользователю, типу действия и типу объекта;
- список клиентов поддерживает фильтрацию клиентов без фотографии, без группы и без актуального оплаченного абонемента;
- для действий редактирования в журнале сохраняются старые и новые значения измененных полей;
- интерфейс корректно работает на ширинах 390 px, 768 px и 1440 px;
- экран отметки посещений удобен для использования тренером с телефона;
- основные экраны не имеют горизонтальной прокрутки страницы и перекрытия элементов.

## 16. Не входит в MVP

В MVP не требуется:

- удаление клиентов;
- удаление групп;
- удаление пользователей;
- хранение способа оплаты;
- интеграция с платежными системами;
- SMS, email или push-уведомления;
- сложные требования к паролю;
- двухфакторная авторизация;
- экспорт отчетов;
- учет филиалов или нескольких залов;
- учет разных помещений внутри зала.
