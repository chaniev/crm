# План реализации MVP Gym CRM для спортивного зала

Дата: 2026-04-17

Основание: [MVP-ТЗ.md](MVP-ТЗ.md), [MVP_SCREEN_MAP.md](MVP_SCREEN_MAP.md)

## 1. Цель плана

План описывает порядок реализации MVP Gym CRM: от подготовки проекта до проверки критериев приемки.

Основной принцип реализации: двигаться вертикальными инкрементами, чтобы после каждого крупного этапа в системе появлялся работающий пользовательский сценарий.

Для frontend экранной моделью реализации служит [MVP_SCREEN_MAP.md](MVP_SCREEN_MAP.md): используется один нормализованный пул экранов MVP, а различия ролей выражаются доступом, составом данных и доступными действиями внутри общих экранов.

## 2. Целевая архитектура MVP

### 2.1. Компоненты

В MVP рекомендуется реализовать 3 основных сервиса:

- `backend`: C#, .NET 10, REST API;
- `frontend`: React;
- `db`: PostgreSQL.

Все сервисы должны запускаться через Docker.

`PostgreSQL` для локальной разработки и для целевого docker-based запуска должен подниматься как отдельный контейнер `db` в `docker-compose`, а не требовать отдельной ручной установки на машине разработчика.

Для локальной разработки нужен `docker-compose.yml`.

### 2.2. Предлагаемая структура проекта

```text
gym-crm/
  backend/
    src/
      GymCrm.Api/
      GymCrm.Application/
      GymCrm.Domain/
      GymCrm.Infrastructure/
    tests/
      GymCrm.Tests/
    Dockerfile
  frontend/
    src/
    Dockerfile
  docker-compose.yml
  docs/
    MVP-ТЗ.md
    IMPLEMENTATION_PLAN.md
    MVP_SCREEN_MAP.md
```

Для MVP допустим более простой вариант backend-монолита, но желательно сразу разделить слои хотя бы логически:

- API: контроллеры, авторизация, DTO;
- Application: бизнес-сценарии и проверки прав;
- Domain: сущности и enum;
- Infrastructure: PostgreSQL, миграции, хеширование паролей, журнал действий.

### 2.3. Frontend UI stack и дизайн-система

Для frontend MVP в качестве базового UI-решения принимается `Mantine`.

Причины выбора:

- проект уже использует `React + Vite`, без `Tailwind`, поэтому `Mantine` проще встроить в текущий каркас без лишней перестройки toolchain;
- для MVP Gym CRM критичны формы, списки, модальные окна, уведомления, выбор дат и адаптивный layout, а эти сценарии хорошо закрываются экосистемой `Mantine`;
- для роли `Coach` ключевой экран посещений должен быть удобным на телефоне, а `Mantine` позволяет быстро собрать единый responsive shell и переиспользуемые формы;
- на этапе MVP важнее скорость разработки, консистентность и читаемость интерфейса, чем построение полностью кастомной component registry с нуля.

Базовый frontend stack:

- `@mantine/core` и `@mantine/hooks` как основа layout и прикладных компонентов;
- `@mantine/form` для форм клиентов, групп, пользователей и смены пароля;
- `@mantine/dates` для выбора дат и связанных сценариев;
- `@mantine/notifications` для пользовательских сообщений об успешных и ошибочных действиях;
- `@tabler/icons-react` как единый набор иконок;
- `Onest` как основной UI-шрифт интерфейса.

Правила дизайн-системы MVP:

- использовать единый светлый theme для MVP; отдельная dark theme в объем MVP не входит;
- строить интерфейс на семантических токенах, а не на разрозненных page-level цветах;
- различия ролей выражать доступом, данными и действиями, а не отдельными цветовыми схемами;
- переиспользовать общие scaffold-компоненты: `AppShell`, layout списков, form sections, карточки, panel с ошибками и пустыми состояниями;
- при кастомной стилизации опираться на theme и CSS variables, а не смешивать несколько UI библиотек в одном экране.

Рекомендуемые базовые токены:

- `primary`: `#145A52`;
- `accent`: `#D89A3D`;
- `background`: `#F6F4EE`;
- `surface`: `#FFFFFF`;
- `text`: `#17312D`;
- `muted`: `#66756F`;
- `border`: `#D7E0DA`;
- `success`: `#2F8F6B`;
- `warning`: `#C98A2E`;

Рекомендуемые параметры интерфейса:

- базовый радиус контролов около `10px`;
- радиус карточек, drawer и modal около `16px`;
- цвет использовать как смысловой сигнал состояния, но не как замену текстовым статусам и предупреждениям;
- для интерфейсов сотрудников избегать визуально "маркетингового" UI и держать акцент на читаемости, плотности данных и скорости сценариев.

### 2.4. Механизм авторизации

Для MVP выбирается cookie-based авторизация.

JWT для браузерного frontend в MVP не используется.

Причины выбора cookie-based авторизации:

- Gym CRM используется как браузерное React-приложение;
- нет требования к публичному API для мобильных приложений или внешних клиентов;
- не нужно хранить JWT в `localStorage` или `sessionStorage`;
- `HttpOnly` cookie снижает риск кражи сессии через XSS;
- подход хорошо поддерживается стандартными средствами ASP.NET Core.

Требования к реализации:

- backend выдает авторизационную cookie после успешного входа;
- cookie должна быть `HttpOnly`;
- cookie должна быть `Secure` в production-режиме;
- cookie должна использовать `SameSite=Lax` при работе frontend и backend в рамках одного сайта;
- logout должен очищать авторизационную cookie;
- frontend должен отправлять запросы к backend с передачей cookie;
- для изменяющих запросов `POST`, `PUT`, `PATCH`, `DELETE` должна быть реализована CSRF-защита;
- CSRF-токен должен передаваться frontend-ом в отдельном HTTP-заголовке, например `X-CSRF-TOKEN`;
- срок жизни сессии рекомендуется установить 8 часов;
- при смене пароля все активные сессии пользователя желательно инвалидировать.

Для локальной разработки рекомендуется запускать frontend и backend через один origin или через dev proxy. Это упрощает работу cookie и CSRF-защиты.

### 2.5. Техническое логирование

Для MVP в качестве основного инструмента технического логирования backend рекомендуется `Serilog.AspNetCore`.

`AuditLog` при этом остается отдельным механизмом бизнес-аудита и не заменяется техническими логами приложения.

Причины выбора:

- проект уже использует `ASP.NET Core`, а `Serilog.AspNetCore` встраивается в стандартный `ILogger`-pipeline без замены прикладного кода на собственный API логирования;
- для MVP нужны технические логи в файлах backend с понятной ротацией и хранением, чтобы диагностика не зависела только от потока контейнерных логов;
- `Serilog` дает request logging middleware с одной агрегированной записью на HTTP-запрос вместо шумного набора событий по умолчанию;
- экосистема `Serilog` позволяет настроить файловое хранение логов с ограничением размера, сроком хранения и предсказуемым именованием файлов.

Принципы реализации:

- backend сохраняет технические логи в файлы в отдельной директории backend;
- директория логов подключается как Docker volume, чтобы файлы логов не терялись при перезапуске контейнера;
- максимальный размер одного файла лога составляет `20 MB`, после чего создается новый файл;
- файлы логов хранятся `10` дней, после чего старые файлы автоматически удаляются;
- имя файла лога должно содержать дату и время его создания;
- для HTTP-запросов используется одна итоговая запись с методом, путем, статусом и длительностью;
- технические логи не дублируют `AuditLog`, а фиксируют состояние приложения, ошибки интеграций, запуск, миграции и HTTP-boundary;
- в технические логи нельзя писать пароль, хеш пароля, auth-cookie, `CSRF`-токены и лишние персональные данные клиентов;
- если понадобится UI для чтения технических логов в локальной среде или на сервере, предпочтительный следующий шаг — подключение отдельного log viewer поверх файловых логов или приемника `Serilog`, а не замена самого логирующего инструмента;
- `OpenTelemetry` стоит рассматривать позже, если появится отдельная задача на полноценную observability-схему с трассировками, метриками и collector-based маршрутизацией.

## 3. Основные сущности данных

### 3.1. Пользователи

Сущность `User`:

- `Id`;
- `FullName`;
- `Login`;
- `PasswordHash`;
- `Role`: `HeadCoach`, `Administrator`, `Coach`;

- пароль хранится только в виде хеша.

- `LastName`;
- `FirstName`;
- `MiddleName`;
- `Phone`;
- `PhotoPath`;
- `PhotoContentType`;
- `PhotoSizeBytes`;
- `PhotoUploadedAt`;

- `UpdatedAt`.

Контактное лицо клиента `ClientContact`:

- `Id`;
- `ClientId`;
- `Type`;
- `FullName`;
- `Phone`.

Ограничение: у клиента не более 2 контактных лиц.

### 3.3. Абонементы клиентов

Сущность `ClientMembership` хранит текущий абонемент клиента и историю изменений абонемента.

Пользователь не создает абонемент отдельным сценарием. В интерфейсе данные `ClientMembership` редактируются как блок в карточке клиента.

Сущность `ClientMembership`:

- `Id`;
- `ClientId`;
- `MembershipType`: `SingleVisit`, `Monthly`, `Yearly`;
- `PurchaseDate`;
- `ExpirationDate`;
- `PaymentAmount`;
- `IsPaid`;
- `SingleVisitUsed`;
- `PaidByUserId`;
- `PaidAt`;
- `ValidFrom`;
- `ValidTo`;
- `ChangeReason`: `NewPurchase`, `Renewal`, `Correction`, `PaymentUpdate`, `SingleVisitWriteOff`;
- `ChangedByUserId`;
- `CreatedAt`.

Правила:

- текущей считается запись, у которой `ValidTo` не заполнено;
- у одного клиента может быть только одна текущая запись `ClientMembership`;
- это правило должно быть закреплено уникальным частичным индексом по `ClientId`, где `ValidTo` не заполнено;
- при изменении типа абонемента, дат, суммы или признака оплаты текущая запись закрывается через `ValidTo`, после чего создается новая текущая запись;
- `ValidFrom` и `ValidTo` описывают срок действия версии записи, а не срок действия самого абонемента;
- срок действия самого абонемента хранится в `ExpirationDate`;
- оформление нового абонемента создает версию с `ChangeReason = NewPurchase`;
- продление текущего абонемента создает версию с `ChangeReason = Renewal`;
- исправление данных текущего абонемента создает версию с `ChangeReason = Correction`;
- отметка оплаты создает версию с `ChangeReason = PaymentUpdate`;
- автоматическое списание разового посещения создает версию с `ChangeReason = SingleVisitWriteOff`;
- изменения дополнительно фиксируются в `AuditLog`.

- `GroupTrainer`: связь тренеров и групп многие-ко-многим.

Сущность `Attendance`:

- `ClientId`;
- `GroupId`;
- `TrainingDate`;
- `IsPresent`;
- `MarkedByUserId`;
- `MarkedAt`;

Ограничение:

### 3.6. Журнал действий

Сущность `AuditLog`:

- `Id`;
- `UserId`;
- `ActionType`;
- `EntityType`;
- `EntityId`;
- `Description`;
- `OldValueJson`;
- `NewValueJson`;
- `CreatedAt`.

Журнал не редактируется через пользовательский интерфейс.

Правила хранения истории изменений:

- `AuditLog` хранит неизменяемую историю действий пользователей;
- для создания объекта `OldValueJson` пустой, `NewValueJson` содержит созданные значения;
- для редактирования объекта `OldValueJson` содержит измененные поля до изменения, `NewValueJson` содержит измененные поля после изменения;
- для архивирования и возврата из архива сохраняется старый и новый статус;
- для абонементов полная история состояния хранится в версиях `ClientMembership`, а `AuditLog` хранит факт действия, измененные поля и `ChangeReason`;
- для изменения посещения сохраняется старое и новое значение признака посещения;
- пароль и хеш пароля не записываются в `OldValueJson` и `NewValueJson`.

- фильтры по группе, статусу, оплате и сроку окончания абонемента;
- фильтр клиентов без фотографии;
- фильтр клиентов без группы;
- фильтр клиентов без актуального оплаченного абонемента.

Задачи frontend:

- фильтры и поиск на общем экране списка клиентов;
- для роли `Coach` список клиентов остается только просмотром без элементов отметки посещения.

Результат:

- клиентская база поддерживает быстрый поиск и отбор записей по основным параметрам.

Критерии готовности:

- поиск по ФИО работает;
- поиск по телефону доступен только Главному тренеру и Администратору;
- фильтры по группе, статусу, оплате и сроку окончания абонемента работают корректно;
- фильтр клиентов без фотографии работает корректно;
- фильтр клиентов без группы работает корректно;
- фильтр клиентов без актуального оплаченного абонемента работает корректно.

### Этап 7. Отметка посещений

Цель: реализовать основной рабочий сценарий тренера.

Задачи backend:

- API списка доступных групп для отметки посещения;
- API получения клиентов группы на дату;
- API сохранения отметок посещения;
- API редактирования посещений задним числом;
- проверка доступа тренера к группе;
- автоматическое списание разового абонемента после посещения;
- расчет предупреждений по абонементу;
- запись отметок и изменений в журнал.

Задачи frontend:

- общий экран отметки посещений для `HeadCoach` и `Coach`;
- выбор группы;
- выбор даты;
- список клиентов группы;
- чекбокс или переключатель посещения напротив клиента;
- отображение предупреждения по абонементу;
- отображение иконки отсутствия оплаты рядом с клиентом при неоплаченном абонементе;
- пустое состояние для тренера без назначенных групп с сообщением о том, что назначенные группы отсутствуют;
- не выводить элементы отметки посещения на экране списка клиентов;
- мобильная адаптация экрана.

Результат:

- Тренер может быстро отмечать посещения в своих группах;
- Главный тренер может отмечать посещения в любой группе.

Критерии готовности:

- Администратор не может открыть или выполнить отметку посещения;
- Тренер не видит телефоны и контактных лиц;
- элементы отметки посещения доступны только на отдельном экране посещений;
- если у тренера нет назначенных групп, экран посещений показывает пустое состояние с понятным сообщением;
- при неактуальном абонементе система предупреждает, но разрешает отметку;
- при неоплаченном абонементе рядом с клиентом отображается иконка отсутствия оплаты;
- разовый абонемент становится использованным после первого посещения.

### Этап 8. История посещений клиента и ролевые ограничения карточки

Цель: добавить в уже реализованную карточку клиента историю посещений и ролевые ограничения отображения.

Задачи backend:

- API истории посещений клиента с пагинацией или иным механизмом частичной загрузки;
- доработать API карточки клиента для ролевого представления;
- ограничение видимости для тренера только по назначенным группам;
- скрытие персональных данных для тренера;
- скрытие для тренера полных данных абонемента, суммы оплаты и подробностей оплаты.

Задачи frontend:

- добавить в существующую карточку клиента блок истории посещений;
- реализовать разные представления данных для Главного тренера, Администратора и Тренера внутри одной карточки клиента;
- не создавать отдельный экран истории посещений.

Результат:

- в существующей карточке клиента доступна история посещений по правам;
- Главный тренер и Администратор видят полную карточку клиента;
- Тренер видит только разрешенный минимум по своим группам.

Критерии готовности:

- тренер не получает через API телефон клиента и контактных лиц;
- тренер не получает через API полные данные абонемента, сумму оплаты и подробности оплаты;
- для тренера история посещений доступна только как блок внутри карточки клиента;
- история посещений отображается согласно правам роли.

### Этап 9. Главная страница

Цель: реализовать стартовый экран с клиентами, у которых скоро закончится абонемент.

Задачи backend:

- API списка клиентов, у которых до окончания абонемента менее 10 дней;
- учитывать только клиентов, для которых дата окончания заполнена;
- добавить сортировку по дате окончания.

Задачи frontend:

- главная страница после входа для `HeadCoach` и `Administrator`;
- список клиентов с истекающими абонементами;
- отображение ФИО, типа абонемента, даты окончания, дней до окончания и признака оплаты.

Результат:

- Главный тренер и Администратор видят клиентов с ближайшим окончанием абонемента.
- после реализации этапа `Главная` становится стартовым экраном для `HeadCoach` и `Administrator`.

Критерии готовности:

- клиенты с окончанием через 10 и более дней не попадают в список;
- экран не разрастается в общий дашборд со вторичными виджетами;
- экран доступен только `HeadCoach` и `Administrator`.

### Этап 10. Журнал действий

Цель: завершить аудит ключевых действий и реализовать API/UI просмотра журнала.

Задачи backend:

- запись входа в систему;

- запись создания и редактирования клиентов;
- запись перевода клиента в архив и обратно;

- запись назначения тренера в группу;
- запись создания и редактирования пользователей;
- запись оформления нового абонемента;
- запись продления абонемента;
- запись исправления данных абонемента;
- запись изменения типа абонемента, дат, суммы и оплаты;
- запись отметки и изменения посещения;
- сохранение старых и новых значений измененных полей в `OldValueJson` и `NewValueJson`;
- исключение пароля и хеша пароля из истории изменений;
- API просмотра журнала с пагинацией и фильтрами.

Фильтры API журнала:

- дата с;
- дата по;
- пользователь;
- тип действия;
- тип объекта.

Задачи frontend:

- экран журнала действий;
- фильтр по пользователю;
- фильтр по типу действия;
- фильтр по типу объекта;
- фильтр по периоду;
- просмотр краткого описания действия;
- просмотр старых и новых значений измененных полей.

Результат:

- Главный тренер и Администратор могут просматривать журнал действий;
- Тренер не имеет доступа к журналу;
- действия, начавшие записываться с ранних этапов, доступны для просмотра через UI журнала.

Критерии готовности:

- ключевые действия из ТЗ фиксируются;
- записи журнала нельзя изменить через UI;
- журнал поддерживает пагинацию;
- журнал фильтруется по дате, пользователю, типу действия и типу объекта;
- история изменений хранит старые и новые значения измененных полей.

### Этап 11. Адаптивный интерфейс и UX-доработки

Цель: привести интерфейс к удобному состоянию для реальной работы.

Задачи:

- настроить общую навигацию по ролям в соответствии с `MVP_SCREEN_MAP.md`;

- закрепить использование семантических токенов `primary`, `accent`, `background`, `surface`, `text`, `muted`, `border`, `success`, `warning`, `danger`;
- убрать точечные цветовые исключения и несогласованные компоненты, если они появились по ходу ранних этапов;
- добавить состояния загрузки и ошибок;
- добавить подтверждения важных действий;
- реализовать автоматическое обновление данных на frontend после успешных CRUD-операций;
- обновлять текущие списки и карточки без обязательной кнопки ручного обновления;
- реализовать мобильное представление для списков и таблиц;

- проверить отсутствие горизонтальной прокрутки страницы на основных экранах.

Результат:

- основные сценарии выполняются без технической помощи.

Критерии готовности:

- тренер может отметить посещения с телефона;
- формы клиента, группы и пользователя удобны на мобильном экране;
- списки клиентов, групп, пользователей и журнал действий имеют корректное мобильное представление;

- после успешных CRUD-операций данные на экране обновляются автоматически;
- основные экраны проверены на ширинах 390 px, 768 px и 1440 px;
- на основных экранах нет горизонтальной прокрутки страницы и перекрытия элементов;
- ошибки backend отображаются понятным образом.

### Этап 12. Тестирование

Цель: проверить бизнес-правила и критерии приемки MVP.

Минимальный набор backend-тестов:

- авторизация;
- cookie-based авторизация и CSRF-защита;
- обязательная смена пароля;
- создание первого Главного тренера;
- матрица прав ролей;
- запрет изменения логина;
- загрузка фотографии клиента в допустимом формате;
- загрузка фотографии клиента в формате HEIC или HEIF;
- корректная конвертация HEIC или HEIF в веб-совместимый формат;
- отклонение фотографии клиента в недопустимом формате;
- отклонение фотографии клиента размером больше 10 MB;
- клиент в нескольких группах;
- группа с несколькими тренерами;
- расчет даты окончания нового месячного абонемента;
- расчет даты окончания нового годового абонемента;
- расчет даты окончания при обычном продлении месячного или годового абонемента от текущей даты окончания;
- расчет даты окончания при продлении месячного или годового абонемента от даты оплаты, если с момента истечения предыдущего абонемента клиент ни разу не посещал занятия в течение одного месяца;
- наличие только одной текущей записи `ClientMembership` у клиента;
- создание новой версии `ClientMembership` при изменении абонемента или оплаты;
- корректная запись `ChangeReason` в `ClientMembership`;
- сохранение старых и новых значений в `AuditLog`;
- фильтрация журнала по дате, пользователю, типу действия и типу объекта;
- списание разового абонемента;
- разрешение посещения при проблемном абонементе;
- отображение иконки отсутствия оплаты для неоплаченного абонемента;
- ограничения доступа тренера;
- запись действий в журнал.

Минимальный набор frontend/e2e-проверок:

- вход и смена пароля;
- создание клиента;
- создание группы и назначение тренеров;
- отметка посещения тренером;
- запрет доступа к чужой группе для тренера;
- просмотр главной страницы;
- просмотр журнала Главным тренером или Администратором;
- проверка основных экранов на ширинах 390 px, 768 px и 1440 px;
- проверка мобильного сценария отметки посещений тренером.
- проверка автоматического обновления данных после CRUD-операций.

Результат:

- основные бизнес-правила защищены тестами;
- критические сценарии проверены вручную или e2e-тестами.

Критерии готовности:

- тесты проходят локально;
- ручной чек-лист MVP выполнен без блокирующих ошибок.

### Этап 13. Docker и подготовка к развертыванию

Цель: подготовить MVP к запуску вне машины разработчика.

Задачи:

- финализировать Dockerfile backend;
- финализировать Dockerfile frontend;
- настроить production-like `docker-compose`;
- убедиться, что PostgreSQL запускается в Docker и в production-like конфигурации не зависит от локально установленной БД;
- настроить Docker volume для фотографий клиентов;
- настроить Docker volume для данных PostgreSQL;
- настроить Docker volume для файлов технических логов backend;
- вынести секреты в переменные окружения;
- описать порядок первого запуска;
- описать порядок применения миграций;
- включить сохранение технических логов backend в файлы;
- настроить ограничение размера файла логов не более `20 MB`;
- настроить автоматическое удаление файлов логов старше `10` дней;
- настроить имя файла логов с датой и временем его создания;
- проверить, что HTTP-запросы пишутся одной итоговой записью с кодом ответа и временем выполнения;
- проверить создание первого пользователя;
- проверить перезапуск контейнеров без потери данных PostgreSQL;
- проверить перезапуск контейнеров без потери загруженных фотографий клиентов;
- проверить перезапуск контейнеров без потери файлов логов в пределах срока хранения.

Результат:

- MVP можно развернуть через Docker;
- данные базы сохраняются между перезапусками;
- технические логи backend сохраняются в файлы и пригодны для диагностики.

Критерии готовности:

- чистый запуск создает базу, применяет миграции и создает первого Главного тренера;
- повторный запуск не дублирует первого пользователя;
- загруженные фотографии клиентов сохраняются после перезапуска контейнеров;
- frontend успешно работает с backend внутри Docker-сети;
- файлы технических логов создаются в выделенной директории backend;
- размер каждого файла логов не превышает `20 MB`;
- файлы логов старше `10` дней автоматически удаляются;
- имя файла логов содержит дату и время его создания;
- технические логи не содержат пароль, хеш пароля, cookie и `CSRF`-токены.

## 7. Риски и решения

| Риск | Что может пойти не так | Как снизить риск |
|---|---|---|
| Неверная модель абонемента | Абонемент случайно начнет проектироваться как отдельный пользовательский сценарий | В UI редактировать абонемент только в карточке клиента, а внутри хранить версии в `ClientMembership` и аудит в `AuditLog` |
| Раздвоение экранов по ролям | Появятся отдельные дубли `Списка клиентов`, `Карточки клиента` и `Экрана посещений`, что раздует MVP | Держаться нормализованного пула экранов из `MVP_SCREEN_MAP.md`, а различия ролей реализовывать через данные и действия внутри общих экранов |
| Разнородный frontend UI stack | В проект попадут несколько несогласованных библиотек, иконок и локальных стилей, из-за чего интерфейс станет рваным и дорогим в поддержке | Зафиксировать `Mantine` как базовый UI kit, `Tabler Icons` как единый набор иконок, `Onest` как основной шрифт и строить кастомизацию через theme и семантические токены |
| Утечка персональных данных тренеру | Тренер увидит телефоны или контакты через API | Проверять права на backend и отдавать разные DTO по ролям |
| Сложная отметка посещений | Тренеру неудобно пользоваться экраном с телефона | Делать экран посещений отдельным мобильным приоритетом |
| Ошибки прав доступа | UI скроет кнопку, но API останется доступным | Все права проверять на backend и покрыть тестами |
| Рост списка клиентов | 10 000 клиентов будут загружаться одним запросом | Сразу реализовать пагинацию, поиск и индексы |
| Неполный аудит | Часть изменений не попадет в журнал | Использовать единый сервис аудита для бизнес-сценариев |

## 8. Минимальный ручной чек-лист перед сдачей MVP

- Запустить проект через Docker.
- Войти первым Главным тренером с паролем `12345678`.
- Сменить пароль при первом входе.
- Создать Администратора.
- Создать Тренера.
- Создать группу и назначить в нее Тренера.
- Создать клиента с двумя контактными лицами.
- Проверить, что клиента можно создать без абонемента.
- Загрузить клиенту фотографию в формате JPEG, PNG, WebP, HEIC или HEIF.
- Проверить, что фотография HEIC или HEIF успешно конвертируется и отображается в системе.
- Проверить, что файл недопустимого формата отклоняется.
- Проверить, что файл больше 10 MB отклоняется.
- Добавить клиента в несколько групп.
- Добавить абонемент позже через карточку клиента.
- Выбрать клиенту месячный абонемент и проверить дату окончания.
- Продлить месячный абонемент и проверить, что при обычном сценарии срок добавляется к текущей дате окончания.
- Проверить, что если с момента истечения предыдущего абонемента клиент ни разу не посещал занятия в течение одного месяца, новый срок считается от даты оплаты.
- Выбрать клиенту годовой абонемент и проверить дату окончания.
- Выбрать клиенту разовый абонемент и отметить посещение.
- Проверить, что разовый абонемент стал использованным.
- Войти Тренером и увидеть только назначенную группу.
- Отметить посещение Тренером.
- Проверить, что Тренер видит фотографии клиентов в назначенной группе.
- Проверить, что в списке клиентов у Тренера нет элементов отметки посещения.
- Проверить, что Тренер не видит телефоны и контактных лиц.
- Проверить, что история посещений для Тренера доступна только внутри карточки клиента.
- Проверить, что при неоплаченном абонементе рядом с клиентом отображается иконка отсутствия оплаты.
- Войти Администратором и отметить оплату клиента.
- Проверить, что Администратор не может отмечать посещение.
- Проверить список клиентов с абонементом, истекающим менее чем через 10 дней.
- Проверить журнал действий Главным тренером.
- Проверить журнал действий Администратором.
- Проверить запрет журнала для Тренера.
- Проверить фильтры журнала по периоду, пользователю, типу действия и типу объекта.
- Проверить, что при редактировании в журнале видны старые и новые значения измененных полей.
- Проверить, что после создания, редактирования, архивирования и других CRUD-операций данные обновляются без ручного нажатия кнопки "Обновить".
- Проверить основные экраны на ширинах 390 px, 768 px и 1440 px.
- Проверить, что на основных экранах нет горизонтальной прокрутки страницы.
- Проверить, что экран отметки посещений удобен для работы с телефона.

## 9. Результат реализации MVP

По завершении плана должна получиться Gym CRM, в которой:

- есть вход по логину и паролю;
- роли работают согласно матрице прав;
- frontend следует нормализованному пулу экранов из `MVP_SCREEN_MAP.md`, а ролевые различия реализованы через данные и доступные действия внутри общих экранов;
- frontend использует единый UI stack на базе `Mantine`, `Tabler Icons`, шрифта `Onest` и семантических theme-токенов;
- Главный тренер управляет пользователями;
- Главный тренер и Администратор управляют клиентами и группами;
- клиент может быть в нескольких группах;
- у группы может быть несколько тренеров;
- у клиента в карточке выбран текущий тип абонемента, который хранится в `ClientMembership`;
- посещения отмечаются по группе и дате;
- разовое посещение списывается автоматически;
- предупреждения по абонементу не блокируют отметку посещения;
- действия пользователей попадают в журнал;
- backend сохраняет технические логи в файлы по заданным правилам хранения и ротации;
- интерфейс работает на мобильных устройствах;
- основные экраны корректно отображаются на ширинах 390 px, 768 px и 1440 px.
