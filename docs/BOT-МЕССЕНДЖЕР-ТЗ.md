# ТЗ на бот для Gym CRM в мессенджерах

Дата: 2026-04-24

Основание: [MVP-ТЗ.md](MVP-ТЗ.md), [MVP_SCREEN_MAP.md](MVP_SCREEN_MAP.md), текущая реализация Gym CRM.

## 1. Назначение

Бот предназначен для быстрых операционных действий в Gym CRM через мессенджер без замены основного web-интерфейса.

Бот должен быть реализован для одного или нескольких поддерживаемых мессенджеров:

- `Telegram`;
- `MAX`.

Конкретный мессенджер выбирается при внедрении. Бизнес-логика бота не должна зависеть от конкретной платформы: интеграция с мессенджером должна быть вынесена в платформенный адаптер.

## 2. Цели

Бот должен закрывать сценарии, где пользователю удобнее выполнить короткое действие с телефона:

- отметить посещаемость группы;
- быстро найти клиента;
- получить список клиентов с заканчивающимися или неоплаченными абонементами;
- отметить оплату текущего абонемента;
- загрузить фотографию клиента;
- получить служебные напоминания и краткие сводки.

Основной источник данных и прав доступа остается в Gym CRM backend.

## 3. Роли пользователей

Бот должен учитывать существующие роли CRM:

- `HeadCoach` — главный тренер;
- `Administrator` — администратор;
- `Coach` — тренер.

Права в боте должны соответствовать правам в web-интерфейсе и backend API.

### 3.1. Ограничения роли `Coach`

Тренер в боте должен видеть только клиентов групп, где он назначен тренером.

Тренеру разрешено видеть:

- фотографию клиента;
- ФИО клиента;
- доступные тренеру группы клиента;
- элементы отметки посещения;
- предупреждение о проблеме с абонементом без раскрытия полных данных;
- индикатор отсутствия оплаты без суммы и платежных деталей.

Тренеру запрещено видеть:

- телефон клиента;
- контактных лиц клиента;
- телефоны контактных лиц;
- полные данные абонемента;
- сумму оплаты;
- полные данные оплаты;
- журнал действий;
- административные команды.

## 4. Поддерживаемые платформы

Бот должен иметь платформенно-независимое ядро и адаптеры мессенджеров.

Минимальные возможности адаптера:

- прием входящих сообщений и нажатий на кнопки;
- отправка текстовых сообщений;
- отправка кнопок выбора действия;
- получение файлов изображений от пользователя;
- идентификация пользователя и чата в рамках выбранной платформы;
- проверка подлинности входящих событий, если это поддерживает платформа;
- обработка повторных или устаревших событий без дублирования действий.

Если выбранный мессенджер не поддерживает часть интерактивных возможностей, сценарий должен иметь текстовую fallback-версию.

## 5. Авторизация и привязка аккаунта

Бот не должен принимать логин и пароль пользователя CRM в чате.

Пользователь должен привязать аккаунт мессенджера к пользователю CRM одним из безопасных способов:

- через одноразовый код, созданный в web-интерфейсе CRM;
- через одноразовую ссылку, созданную в web-интерфейсе CRM;
- через другой согласованный backend-driven flow без передачи пароля в мессенджер.

Требования к привязке:

- одноразовый код или ссылка действуют ограниченное время;
- код или ссылка могут быть использованы только один раз;
- пользователь может отвязать мессенджер от CRM;
- неактивный пользователь CRM не может работать с ботом;
- пользователь с обязательной сменой пароля должен сначала сменить пароль в web-интерфейсе;
- при смене роли, деактивации пользователя или изменении назначенных групп бот должен применять актуальные права без ручной синхронизации.

## 6. Основные сценарии

### 6.1. Старт и меню

При первом обращении бот должен:

- определить, привязан ли аккаунт мессенджера к пользователю CRM;
- если привязки нет, предложить выполнить привязку;
- если привязка есть, показать меню доступных действий с учетом роли пользователя.

Минимальные разделы меню:

- `Посещения`;
- `Поиск клиента`;
- `Заканчивающиеся абонементы`;
- `Неоплаченные абонементы`;
- `Сводка`;
- `Настройки`;
- `Отвязать аккаунт`.

Недоступные по роли разделы не должны отображаться и не должны выполняться при ручном вводе команды.

### 6.2. Отметка посещаемости

Сценарий доступен:

- `HeadCoach` — по всем группам;
- `Coach` — только по назначенным группам.

Сценарий должен позволять:

- выбрать дату, по умолчанию текущую дату;
- выбрать группу из доступных пользователю;
- получить список клиентов группы;
- отметить каждого клиента как `был` или `не был`;
- сохранить отметки одним действием;
- изменить отметки за выбранную дату, если это разрешено ролью и backend-политиками;
- увидеть предупреждение по абонементу клиента без нарушения ролевых ограничений.

После сохранения бот должен показать краткий результат:

- название группы;
- дату;
- количество отмеченных клиентов;
- количество присутствовавших;
- количество отсутствовавших;
- список клиентов с предупреждениями по абонементу или оплате, если доступно роли.

Если в группе нет клиентов, бот должен показать пустое состояние.

Если у тренера нет назначенных групп, бот должен показать сообщение о том, что назначенные группы отсутствуют.

### 6.3. Быстрый поиск клиента

Сценарий доступен всем ролям с учетом ролевого среза данных.

Поиск должен поддерживать:

- поиск по ФИО;
- поиск по телефону только для `HeadCoach` и `Administrator`.

Результат поиска должен показывать список найденных клиентов. При выборе клиента бот должен показать карточку в формате, разрешенном роли пользователя.

Для `HeadCoach` и `Administrator` карточка может содержать:

- ФИО;
- телефон;
- группы;
- статус клиента;
- текущий абонемент;
- признак оплаты;
- дату окончания абонемента;
- фотографию, если она есть.

Для `Coach` карточка должна содержать только безопасный набор данных:

- фотография;
- ФИО;
- доступные тренеру группы;
- предупреждение о проблеме с абонементом;
- индикатор отсутствия оплаты;
- краткая история посещений по доступным группам, если сценарий реализован.

### 6.4. Заканчивающиеся абонементы

Сценарий доступен:

- `HeadCoach`;
- `Administrator`.

Бот должен показывать клиентов с текущим абонементом, который заканчивается менее чем через 10 дней.

Для каждого клиента в списке нужно показать:

- ФИО;
- тип абонемента;
- дату окончания;
- количество дней до окончания;
- признак оплаты.

Из списка пользователь должен иметь возможность открыть краткую карточку клиента.

### 6.5. Неоплаченные абонементы

Сценарий доступен:

- `HeadCoach`;
- `Administrator`.

Бот должен показывать клиентов с текущим неоплаченным абонементом.

Для каждого клиента нужно показать:

- ФИО;
- тип абонемента;
- дату покупки;
- дату окончания, если она есть;
- текущий статус оплаты.

Из списка пользователь должен иметь возможность:

- открыть краткую карточку клиента;
- перейти к отметке оплаты.

### 6.6. Отметка оплаты

Сценарий доступен:

- `HeadCoach`;
- `Administrator`.

Бот должен позволять отметить текущий абонемент клиента как оплаченный.

Требования:

- действие должно требовать явного подтверждения;
- перед подтверждением бот должен показать ФИО клиента и краткое описание текущего абонемента;
- после успешного действия бот должен показать результат;
- действие должно записываться в журнал действий CRM;
- если текущий абонемент уже оплачен, бот должен показать актуальный статус и не выполнять повторную запись.

Создание, продление и исправление абонемента через бот в MVP не требуется.

### 6.7. Загрузка фотографии клиента

Сценарий доступен:

- `HeadCoach`;
- `Administrator`.

Бот должен позволять:

- найти клиента;
- отправить фотографию в чат;
- прикрепить фотографию к карточке выбранного клиента;
- заменить существующую фотографию после подтверждения.

Файл должен проходить те же проверки, что и загрузка фото через web/API:

- допустимый тип файла;
- допустимый размер;
- обработка и конвертация форматов, если это поддерживается текущим backend;
- сохранение метаданных фотографии.

Если пользователь отправил фото без выбранного клиента, бот должен попросить сначала выбрать клиента.

### 6.8. Напоминания тренеру

Сценарий доступен:

- `HeadCoach`;
- `Coach`.

Бот должен уметь отправлять напоминание перед тренировкой.

Минимальное содержание напоминания:

- название группы;
- время тренировки;
- количество клиентов;
- кнопка или команда для перехода к отметке посещаемости.

Время отправки напоминаний должно настраиваться конфигурацией или пользовательскими настройками.

### 6.9. Сводка для главного тренера

Сценарий доступен:

- `HeadCoach`.

Бот должен уметь отправлять ежедневную сводку.

Минимальное содержание сводки:

- количество тренировок на текущий день;
- количество групп, где посещаемость уже отмечена;
- количество групп, где посещаемость еще не отмечена;
- количество клиентов с заканчивающимися абонементами;
- количество клиентов с неоплаченными абонементами;
- краткий список важных событий или ссылка/кнопка на web-интерфейс журнала действий.

## 7. Уведомления

Бот должен поддерживать плановые и событийные уведомления.

Минимальный набор уведомлений:

- напоминание тренеру перед занятием;
- ежедневная сводка главному тренеру;
- уведомление администратору или главному тренеру о заканчивающихся абонементах;
- уведомление администратору или главному тренеру о неоплаченных абонементах;
- уведомление о неотмеченной посещаемости после окончания тренировки.

Пользователь должен иметь возможность отключить уведомления бота для своего аккаунта.

Системные уведомления не должны раскрывать данные, недоступные роли пользователя.

## 8. Команды и навигация

Бот должен поддерживать команды или кнопочное меню. Конкретная форма зависит от выбранного мессенджера.

Рекомендуемые команды:

- `start` — старт, проверка привязки и показ меню;
- `help` — краткая справка;
- `today` — группы и тренировки на сегодня;
- `attendance` — отметка посещаемости;
- `client` — поиск клиента;
- `expiring` — заканчивающиеся абонементы;
- `unpaid` — неоплаченные абонементы;
- `summary` — сводка;
- `settings` — настройки уведомлений;
- `unlink` — отвязка аккаунта мессенджера.

Команды должны быть доступны также через кнопки, если выбранный мессенджер это поддерживает.

## 9. Аудит

Все действия, изменяющие данные CRM, должны записываться в журнал действий.

Минимально аудируемые действия:

- привязка аккаунта мессенджера;
- отвязка аккаунта мессенджера;
- сохранение посещаемости;
- отметка оплаты;
- загрузка или замена фотографии клиента;
- ошибки доступа при попытке выполнить запрещенное действие.

В записи аудита нужно сохранять:

- пользователя CRM;
- действие;
- тип и идентификатор сущности;
- дату и время;
- источник действия: `Bot`;
- платформу мессенджера: `Telegram` или `MAX`;
- старое и новое значение для изменяющих действий, если применимо.

Идентификаторы пользователя и чата мессенджера не должны выводиться в пользовательские экраны журнала без необходимости.

## 10. Архитектурные требования

### 10.1. Общий подход

Рекомендуемая архитектура:

- `Bot Core` — платформенно-независимая логика сценариев;
- `Messenger Adapter` — адаптер конкретного мессенджера;
- `CRM Integration` — слой доступа к backend-сценариям CRM;
- `Bot Storage` — хранение привязок аккаунтов, состояния диалогов и настроек уведомлений;
- `Scheduler` — плановые напоминания и сводки.

Бот не должен работать с базой CRM напрямую в обход доменной логики и проверок прав.

### 10.2. Платформенный адаптер

Адаптер мессенджера должен отвечать только за:

- прием событий платформы;
- нормализацию входящих событий во внутренний формат;
- отправку сообщений;
- отправку кнопок;
- получение файлов;
- преобразование ошибок платформы во внутренние ошибки.

Адаптер не должен содержать бизнес-правила CRM.

### 10.3. Состояние диалогов

Бот должен хранить состояние многошаговых сценариев:

- выбранная дата посещаемости;
- выбранная группа;
- выбранный клиент;
- ожидаемое действие пользователя;
- временные данные подтверждения.

Состояние должно иметь срок жизни. При истечении срока бот должен безопасно завершить сценарий и предложить начать заново.

### 10.4. Интеграция с CRM

Для реализации сценариев бот должен переиспользовать существующие backend-сценарии или отдельные internal API, которые применяют те же правила доступа.

Переиспользуемые области CRM:

- пользователи и роли;
- доступные пользователю группы;
- список клиентов;
- карточка клиента;
- фотографии клиентов;
- текущий абонемент;
- заканчивающиеся абонементы;
- неоплаченные абонементы;
- посещаемость;
- аудит.

Если существующих endpoint'ов недостаточно, нужно добавить backend API без дублирования бизнес-логики в коде бота.

## 11. Данные

Для бота требуется хранить дополнительные данные.

### 11.1. Привязка аккаунта мессенджера

Минимальные поля:

- идентификатор записи;
- идентификатор пользователя CRM;
- платформа: `Telegram` или `MAX`;
- идентификатор пользователя платформы;
- идентификатор чата платформы, если требуется;
- дата привязки;
- дата последнего взаимодействия;
- дата отвязки, если аккаунт отвязан;
- признак активности.

### 11.2. Одноразовый код привязки

Минимальные поля:

- идентификатор записи;
- идентификатор пользователя CRM;
- хеш кода или токена;
- дата создания;
- дата истечения;
- дата использования;
- признак использования.

Код или токен привязки нельзя хранить в открытом виде.

### 11.3. Настройки уведомлений

Минимальные поля:

- идентификатор пользователя CRM;
- платформа;
- включены ли уведомления;
- время ежедневной сводки;
- время напоминания перед тренировкой;
- дата последнего успешного уведомления.

### 11.4. Состояние диалога

Минимальные поля:

- платформа;
- идентификатор чата;
- текущий сценарий;
- сериализованные данные сценария;
- дата создания;
- дата обновления;
- дата истечения.

## 12. Безопасность

Требования:

- не принимать и не хранить пароль CRM в боте;
- не отправлять секреты и токены в сообщения;
- не логировать персональные данные сверх необходимого для диагностики;
- проверять подлинность входящих событий от мессенджера, если платформа это поддерживает;
- ограничивать частоту команд и попыток привязки;
- не раскрывать данные, недоступные роли пользователя;
- инвалидировать доступ при деактивации пользователя CRM;
- учитывать изменения ролей и назначенных групп;
- обрабатывать повторные callback-события идемпотентно;
- не выполнять изменяющее действие без явного подтверждения там, где есть риск ошибки.

## 13. Нефункциональные требования

### 13.1. Надежность

Бот должен:

- корректно обрабатывать недоступность CRM backend;
- показывать пользователю понятное сообщение при временной ошибке;
- повторять отправку плановых уведомлений при временных ошибках платформы;
- не дублировать изменяющие действия при повторной доставке события;
- иметь health-check endpoint или другой способ проверки готовности.

### 13.2. Производительность

Бот должен отвечать на пользовательские команды без заметной задержки при штатной работе backend.

Долгие операции, например загрузка и обработка фотографии, должны сопровождаться промежуточным сообщением о выполнении.

### 13.3. Локализация

Тексты бота должны быть на русском языке.

Формат дат и времени должен соответствовать рабочей таймзоне проекта.

### 13.4. Логирование и мониторинг

Бот должен писать технические логи:

- входящие события;
- исходящие сообщения без лишних персональных данных;
- ошибки платформенного адаптера;
- ошибки интеграции с CRM;
- результат плановых задач.

Логи не должны содержать токены, одноразовые коды, пароли и избыточные персональные данные.

## 14. Развертывание и конфигурация

Бот должен запускаться в составе инфраструктуры проекта или как отдельный сервис рядом с backend.

Конфигурация должна задаваться через переменные окружения или защищенное хранилище:

- включенная платформа;
- токен или секрет платформы;
- публичный base URL для webhook, если используется webhook-режим;
- внутренний URL CRM backend;
- настройки планировщика уведомлений;
- ограничения частоты команд;
- срок жизни одноразовых кодов;
- срок жизни состояния диалога.

Секреты мессенджера нельзя хранить в репозитории.

## 15. Тестирование

Минимальный набор проверок:

- пользователь может привязать аккаунт мессенджера к CRM;
- одноразовый код нельзя использовать повторно;
- неактивный пользователь не может работать с ботом;
- пользователь с обязательной сменой пароля получает требование сменить пароль в web-интерфейсе;
- тренер видит только назначенные группы;
- тренер не видит телефон, контакты, сумму оплаты и полные данные абонемента;
- тренер может отметить посещаемость только в назначенной группе;
- главный тренер может отметить посещаемость в любой группе;
- администратор не получает доступ к сценариям, которые не разрешены его роли;
- главный тренер и администратор видят заканчивающиеся абонементы;
- главный тренер и администратор видят неоплаченные абонементы;
- отметка оплаты требует подтверждения;
- загрузка фотографии требует выбранного клиента;
- изменяющие действия попадают в журнал действий;
- повторное событие от платформы не дублирует изменяющее действие;
- при недоступности CRM backend бот показывает понятную ошибку.

Тесты платформенного адаптера должны использовать fake adapter или mock-события мессенджера, чтобы бизнес-сценарии проверялись без реального Telegram или MAX.

## 16. Этапы реализации

### Этап 1. Технический каркас

- Создать сервис или модуль бота.
- Добавить платформенно-независимое ядро сценариев.
- Добавить интерфейс адаптера мессенджера.
- Реализовать один адаптер для выбранного мессенджера: `Telegram` или `MAX`.
- Добавить хранение привязок аккаунтов.
- Добавить health-check и базовое логирование.

### Этап 2. Авторизация и меню

- Реализовать привязку аккаунта через одноразовый код или ссылку.
- Реализовать отвязку аккаунта.
- Реализовать role-aware главное меню.
- Добавить проверки неактивного пользователя и обязательной смены пароля.

### Этап 3. Посещаемость

- Реализовать выбор даты и группы.
- Реализовать выдачу списка клиентов группы.
- Реализовать отметку `был` / `не был`.
- Реализовать сохранение отметок.
- Добавить аудит действий.
- Покрыть сценарий тестами для `HeadCoach` и `Coach`.

### Этап 4. Поиск клиента

- Реализовать поиск по ФИО.
- Реализовать поиск по телефону для management-ролей.
- Реализовать role-based карточку клиента.
- Проверить отсутствие чувствительных данных у роли `Coach`.

### Этап 5. Абонементы и оплата

- Реализовать список заканчивающихся абонементов.
- Реализовать список неоплаченных абонементов.
- Реализовать отметку оплаты с подтверждением.
- Добавить аудит отметки оплаты.

### Этап 6. Фотографии

- Реализовать сценарий выбора клиента.
- Реализовать прием фотографии из мессенджера.
- Реализовать загрузку фотографии в CRM.
- Реализовать подтверждение замены существующей фотографии.

### Этап 7. Напоминания и сводки

- Реализовать напоминание тренеру перед занятием.
- Реализовать ежедневную сводку главному тренеру.
- Реализовать уведомление о неотмеченной посещаемости.
- Реализовать настройки включения и отключения уведомлений.

## 17. Не входит в MVP бота

В MVP бота не требуется:

- создание клиента;
- полное редактирование клиента;
- архивирование и восстановление клиента;
- создание и редактирование групп;
- назначение тренеров в группы;
- создание и редактирование пользователей;
- создание, продление и исправление абонемента;
- просмотр полного журнала действий;
- смена пароля CRM;
- полноценная замена web-интерфейса.

Эти сценарии должны оставаться в web-интерфейсе CRM.

## 18. Критерии приемки

Бот считается готовым к MVP-приемке, если:

- реализован минимум один платформенный адаптер: `Telegram` или `MAX`;
- бизнес-логика бота не зависит от конкретного мессенджера;
- пользователь может безопасно привязать и отвязать аккаунт;
- меню и команды учитывают роль пользователя;
- тренер не получает доступ к запрещенным данным;
- тренер может отметить посещаемость по назначенной группе;
- главный тренер может отметить посещаемость по любой группе;
- главный тренер и администратор могут посмотреть заканчивающиеся и неоплаченные абонементы;
- главный тренер и администратор могут отметить оплату с подтверждением;
- главный тренер и администратор могут загрузить фото клиента;
- изменяющие действия фиксируются в журнале действий;
- повторные события от мессенджера не создают дубли;
- при ошибках backend или мессенджера пользователь получает понятное сообщение;
- основные сценарии покрыты автоматическими тестами с fake adapter.
