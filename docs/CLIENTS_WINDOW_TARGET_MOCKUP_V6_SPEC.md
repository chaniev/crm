# UX-спецификация V6: экран "Клиенты"

Основание: текущий экран и скриншот показывают три повторяющихся сбоя в задаче тренера или администратора: список читается слишком медленно, правый блок перетягивает на себя роль полноценной карточки, а важное действие теряется среди статусов, бейджей и сервисных блоков.

## 1. Главный сценарий

Пользователь открывает раздел "Клиенты", быстро находит нужного клиента по поиску или quick-фильтрам, сканирует список по рискам и следующему действию, выбирает строку и сразу выполняет одно из трех действий без перехода в карточку:

- отметить визит;
- принять оплату;
- предложить или оформить продление.

Переход в полную карточку нужен только для редактирования, истории и редких сценариев.

## 2. Что уплотнить

- Верхнюю часть экрана: поиск, статус, группа и `Еще фильтры` должны занимать один компактный ряд, quick-фильтры идут второй строкой.
- Таблицу: высота строки 72-80 px, без больших пустых зазоров и без лишних вторых уровней текста в каждой колонке.
- Колонки: оставить только operational-поля для принятия решения в списке:
  `Клиент`, `Статус и абонемент`, `Следующий шаг`, `Группа`, `Последний визит`.
- Правый блок: показывать 4-6 ключевых фактов, один главный alert/action block и 3 последних события. Не строить длинную карточку внутри preview.

## 3. Что убрать или ослабить

- Убрать из списка дубли статуса, если тот же сигнал уже читается из абонемента или next step.
- Ослабить вторичные кнопки в строках. Основной паттерн: выбор строки, а не набор inline-action-кнопок на каждой записи.
- Убрать тяжёлые секции в preview вроде больших "паспортных" карточек, длинных списков свойств и нескольких равносильных CTA.
- Ослабить визуальный вес фото: оно помогает распознаванию, но не должно быть главным объектом экрана.

## 4. Поведение правого предпросмотра

- Preview является контекстной панелью, а не второй страницей.
- На desktop панель sticky и обновляется при выборе строки без потери скролла, поиска, фильтров и позиции в списке.
- Внутри preview всегда есть один главный блок `Нужно сейчас`, который показывает только ближайшее полезное действие.
- Если срочного действия нет, вместо alert-блока показывается нейтральное состояние `Ничего срочного`.
- Кнопка `Открыть полную карточку` ведет в детальный экран; возврат обратно обязан сохранять search, filters, sort и scroll.
- На tablet/mobile preview уходит под список или открывается как drawer снизу/справа, но не ломает основной list-first сценарий.

## 5. Приоритет для реализации

1. Уплотнить toolbar и список.
2. Перестроить правый preview в краткий contextual panel.
3. Оставить один главный CTA по выбранному клиенту.
4. Сохранить состояние списка при входе и возврате из полной карточки.

## 6. Проверка после реализации

- За 5-10 секунд пользователь должен находить клиента и понимать, нужно ли действие.
- Для клиентов с долгом, окончанием абонемента и отсутствием абонемента следующий шаг должен читаться прямо из списка без открытия карточки.
- Выбор строки не должен вызывать layout shift, jump scroll или сброс фильтров.
