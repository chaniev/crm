# TASK-031: Реализовать backend-модель филиалов и залов

## Status
risky

## Goal
Backend становится источником истины для филиалов, залов и всех правил привязки клиентов, групп, секций, тренеров и абонементов к филиалам.

## Context
Задача создана после завершения уточнений в `TASK-022`.

Принятая модель:
- филиал задается в настройках CRM;
- обязательное поле филиала: название;
- дополнительные поля филиала: адрес, описание;
- филиал можно архивировать;
- клиент может быть только в одном филиале;
- группа может быть только в одном филиале;
- секции привязаны к филиалам;
- внутри филиала есть список залов;
- поля зала: название, описание;
- зал нельзя удалить или архивировать, если к нему привязаны группы или расписание;
- у тренера нет прямой связи с филиалами, филиалы тренера определяются через назначенные ему группы;
- тренер видит только клиентов в своих группах;
- абонемент привязан к филиалу клиента;
- при переводе клиента в другой филиал абонемент сохраняется, клиент удаляется из текущих групп и остается без группы, если администратор не выбрал группу из нового филиала;
- миграция существующих данных не нужна, деплой будет с нуля.

## User role
администратор / владелец / тренер

## Problem
Филиалы затрагивают доменную модель, права доступа, валидацию, API contracts и persistence. Эти правила должны жить в backend, а frontend и bot должны только потреблять готовые backend-контракты и ProblemDetails.

## Scope
- Добавить сущность филиала: название, адрес, описание, архивный статус.
- Добавить сущность зала: название, описание, связь с одним филиалом.
- Добавить backend API для администрирования филиалов и залов.
- Добавить связь клиента с одним филиалом.
- Добавить связь группы с одним филиалом и залом этого филиала.
- Добавить связь секции с филиалом, если секции уже выделены как отдельная сущность.
- Не добавлять прямую связь `тренер-филиал`: доступ тренера к филиалам выводится через назначенные ему группы.
- Запретить добавление клиента из филиала A в группу филиала B.
- Запретить назначение тренера на действия с группой, на которую он не назначен.
- Реализовать перевод клиента в другой филиал: сохранить абонемент, удалить клиента из текущих групп, опционально назначить группу из нового филиала.
- Валидировать, что выбранная при переводе группа относится к новому филиалу.
- Привязку абонемента к филиалу считать производной от текущего филиала клиента, если для отчетов не потребуется отдельный backend snapshot.
- Запретить удаление или архивирование зала, если к нему привязаны группы или расписание.
- Обновить ProblemDetails для новых ошибок валидации и прав доступа.
- Обновить backend tests на новые rules и contracts.

## Out of scope
- Frontend-экраны управления филиалами и залами.
- Bot UI/flows.
- Финансовая отчетность по филиалам.
- Миграция существующих production-данных.
- Полная модель расписания, переносов и конфликтов времени.

## Constraints
- Backend владеет roles, permissions, access scope, memberships, attendance, audit semantics и validation semantics.
- Domain не должен зависеть от HTTP/UI.
- Transport-level код не должен содержать доменные правила филиалов.
- Contract changes требуют обновления всех consumers.
- Schema changes требуют migrations, но legacy data migration не нужна из-за деплоя с нуля.

## Acceptance criteria
- [ ] В backend есть сущность филиала с названием, адресом, описанием и архивным статусом.
- [ ] В backend есть сущность зала с названием, описанием и обязательной связью с филиалом.
- [ ] Клиент обязательно относится ровно к одному филиалу.
- [ ] Группа относится ровно к одному филиалу и выбирает зал из этого филиала.
- [ ] Секция относится к филиалу, если секции моделируются отдельно.
- [ ] Нет прямой связи `тренер-филиал`.
- [ ] Тренер получает доступ к клиентам только через назначенные ему группы.
- [ ] Backend запрещает добавить клиента в группу другого филиала.
- [ ] Backend запрещает вести/изменять группу тренеру, который не назначен на нее.
- [ ] Перевод клиента в другой филиал сохраняет абонемент и очищает текущие группы.
- [ ] При переводе клиента администратор может выбрать группу из нового филиала.
- [ ] Backend запрещает выбрать при переводе группу не из нового филиала.
- [ ] Зал нельзя удалить или архивировать, если к нему привязаны группы или расписание.
- [ ] Ошибки прав и валидации возвращаются через согласованные ProblemDetails.
- [ ] Обновлены backend integration/regression tests.
