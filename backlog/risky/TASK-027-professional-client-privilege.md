# TASK-027: Реализовать признак профессионал

## Status
risky

## Goal
Спортсмен из профгруппы с турнирной привилегией может иметь признак `профессионал`, не оплачивать посещения и не отображаться как должник.

## Context
В inbox была короткая заметка: `"Нулевой" абонемент`.

Уточнение от 2026-05-07: спортсмены из профгрупп, отличившиеся на важных турнирах, получают привилегию и могут не оплачивать абонементы за посещения. В оплате это должно быть явно отражено, чтобы клиент не отображался как должник.

Дополнительные уточнения от 2026-05-07:
- должна быть возможность отметить клиента признаком `профессионал`;
- цена абонемента для такого клиента равна 0;
- абонемент считается оплаченным;
- количество посещений не ограничено;
- срок действия длится пока у клиента выставлен признак `профессионал`;
- признак хранится на клиенте;
- абонемент клиента с признаком `профессионал` исключается из выручки;
- при выставлении признака достаточно комментария администратора;
- включать и отключать признак может главный тренер;
- в журнале событий достаточно фиксировать включение/отключение, кто переключил, дату и время;
- прежнее название признака `нулевой абонемент` переименовано в `профессионал`.

## User role
главный тренер / администратор / владелец

## Problem
Сейчас клиент с бесплатной турнирной привилегией может выглядеть как клиент с неоплаченным абонементом. Нужно явно отделить признак `профессионал` от долга: для CRM это действующий оплаченный абонемент с ценой 0 и безлимитными посещениями.

## Scope
- Добавить на клиента признак `профессионал`.
- При включении признака хранить комментарий администратора как основание привилегии.
- Разрешить включение и отключение признака главному тренеру.
- При включенном признаке считать абонемент оплаченным, с ценой 0 и без лимита посещений.
- Исключить клиентов с признаком `профессионал` из должников и списков неоплаченных абонементов.
- Исключить абонементы клиентов с признаком `профессионал` из выручки в финансовых отчетах.
- Фиксировать включение и отключение признака в журнале событий: кто переключил, дату и время.
- Обновить frontend и bot consumers, если backend-контракт меняется.

## Out of scope
- Хранение отдельного турнира, достижения, документа или вложения для основания привилегии.
- Автоматическое назначение признака `профессионал` по результатам турниров.
- Расчет скидок, возвратов или компенсаций.
- Изменение правил обычных платных абонементов.

## Constraints
- Backend владеет memberships, subscriptions, visit write-offs, debtor status, financial report semantics and audit semantics.
- Frontend и bot не должны самостоятельно решать, является ли клиент должником; этот статус должен приходить из backend-контракта.
- Посещения клиента с признаком `профессионал` должны фиксироваться в посещаемости, но не должны исчерпывать абонемент или приводить к долгу.
- Нужно сохранить ProblemDetails contracts для ошибок прав и валидации.
- Contract changes требуют обновления всех потребителей.

## Acceptance criteria
- [ ] В модели клиента есть признак `профессионал`.
- [ ] При включении признака сохраняется комментарий администратора.
- [ ] Только главный тренер может включать и отключать признак.
- [ ] Клиент с включенным признаком имеет цену абонемента 0 и считается оплаченным.
- [ ] Посещения клиента с признаком `профессионал` фиксируются, но не ограничиваются и не списывают лимит.
- [ ] Клиент с признаком `профессионал` не попадает в должники и списки неоплаченных абонементов.
- [ ] Абонементы клиентов с признаком `профессионал` исключаются из выручки в финансовых отчетах.
- [ ] В журнале событий фиксируются включение/отключение признака, actor, дата и время.
- [ ] Frontend отображает признак `профессионал` как льготный/оплаченный статус, а не как неоплаченный абонемент.
- [ ] Bot не показывает клиента с признаком `профессионал` в списке неоплаченных абонементов.
