# TASK-032: Реализовать frontend для филиалов и залов

## Status
risky

## Goal
Администратор управляет филиалами и залами в настройках CRM, а рабочие формы клиентов, групп и переводов клиента используют backend-контракты филиалов без дублирования доменных правил.

## Context
Задача создана после завершения уточнений в `TASK-022` и зависит от backend-контрактов из `TASK-031`.

Принятая модель:
- филиалы задаются в настройках CRM;
- филиал: название, адрес, описание, архивный статус;
- зал: название, описание, принадлежность филиалу;
- клиент может быть только в одном филиале;
- группа может быть только в одном филиале и зале этого филиала;
- при переводе клиента в другой филиал администратор может выбрать группу из нового филиала;
- frontend не должен сам решать, можно ли выполнить cross-branch действие, окончательная валидация остается на backend.

## User role
администратор / владелец / тренер

## Problem
После появления backend-модели филиалов frontend должен дать администраторам понятное управление справочниками и корректно потреблять новые поля в рабочих формах, не перенося CRM domain rules на клиент.

## Scope
- Добавить UI управления филиалами в разделе настроек CRM.
- Добавить создание и редактирование филиала: название, адрес, описание.
- Добавить отображение архивного статуса филиала и действие архивирования.
- Добавить UI списка залов внутри филиала.
- Добавить создание и редактирование зала: название, описание.
- Показать backend-ошибку, если зал нельзя удалить или архивировать из-за привязанных групп или расписания.
- Обновить frontend API contracts/types под филиалы и залы.
- Добавить выбор филиала в форме клиента.
- Добавить сценарий перевода клиента в другой филиал с опциональным выбором группы из нового филиала.
- После перевода без выбора группы показывать клиента без группы.
- Добавить выбор филиала и зала в форме группы.
- Фильтровать список залов в форме группы по выбранному филиалу, используя backend data.
- Обновить списки/карточки клиентов и групп, если backend contract начнет возвращать филиал/зал.
- Сохранять видимость пунктов и действий по backend permissions, не дублируя правила ролей во frontend.

## Out of scope
- Backend domain rules и migrations.
- Bot consumers.
- Финансовая отчетность по филиалам.
- Полная реализация расписания, переносов и conflict UI.
- Изменение общей информационной архитектуры настроек за пределами филиалов и залов.

## Constraints
- Frontend не должен дублировать CRM validation semantics.
- `frontend/lib/api.ts` должен соответствовать backend contracts.
- Значимое UX-изменение в настройках желательно сверить с `ui-designer`.
- Preserve Mantine and Onest.
- Не ломать narrow-screen flows.

## Acceptance criteria
- [ ] В настройках CRM есть список филиалов.
- [ ] Администратор может создать и отредактировать филиал.
- [ ] Администратор может архивировать филиал.
- [ ] В карточке/деталях филиала есть список залов.
- [ ] Администратор может создать и отредактировать зал.
- [ ] UI корректно показывает backend-запрет удаления/архивирования зала с привязками.
- [ ] В форме клиента выбирается филиал.
- [ ] В переводе клиента между филиалами можно выбрать группу из нового филиала или оставить клиента без группы.
- [ ] В форме группы выбираются филиал и зал из выбранного филиала.
- [ ] Frontend не разрешает очевидно некорректный cross-branch выбор в UI, но backend остается source of truth.
- [ ] Обновлены affected screens/components и API types.
- [ ] Mobile/narrow layout не ломается на формах настроек, клиента и группы.

## Test checklist
- [ ] Запустить `cd frontend && npm run lint`.
- [ ] Запустить `cd frontend && npm run build`.
- [ ] При наличии e2e harness проверить основные admin flows для филиалов/залов.
- [ ] Вручную проверить создание клиента, создание группы и перевод клиента между филиалами.
- [ ] Проверить отображение backend ProblemDetails для запрещенных действий.
