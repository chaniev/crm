# План изменений окна Gym CRM

## Цель

Модифицировать окно Gym CRM в соответствии со скриншотом: привести шапку, навигацию, основную карточку, блок "Истекающие абонементы" и пустое состояние к единому визуальному стилю. Одновременно унифицировать общие элементы интерфейса, чтобы не прописывать одинаковую разметку и стили отдельно на каждой вкладке.

## 1. Предпроектная подготовка

1. Проанализировать текущую структуру интерфейса:
   - где описаны layout, header, navigation и страницы;
   - какие компоненты уже переиспользуются;
   - какие стили дублируются на разных вкладках.

2. Определить границы изменений:
   - какие элементы станут общими UI-компонентами;
   - какие изменения относятся только к экрану "Главная";
   - какие страницы могут быть затронуты из-за унификации layout.

3. Зафиксировать визуальные ориентиры по скриншоту:
   - светлый фон страницы;
   - верхняя шапка с логотипом Gym CRM;
   - горизонтальная pill-навигация;
   - большая белая карточка;
   - блок "Истекающие абонементы";
   - empty state с иконкой календаря.

## 2. Проектирование юнит-тестов до разработки

1. До начала реализации описать, какие сценарии должны быть покрыты тестами.

2. Разделить тесты по уровням:
   - общие UI-компоненты;
   - layout и навигация;
   - конкретный экран "Главная";
   - состояния данных: loading, error, empty, data.

3. Подготовить тест-кейсы для `NavigationTabs`:
   - корректно рендерит все вкладки;
   - подсвечивает активную вкладку;
   - использует правильные пути или обработчики перехода;
   - не требует дублирования настроек на каждой странице.

4. Подготовить тест-кейсы для `AppLayout` и `Header`:
   - отображается логотип Gym CRM;
   - отображается профиль пользователя;
   - навигация присутствует на всех страницах внутри общего layout;
   - активная вкладка определяется по текущему маршруту.

5. Подготовить тест-кейсы для `PageCard`:
   - корректно рендерит вложенный контент;
   - применяет общий контейнер страницы;
   - не требует локальных оберток на каждой вкладке.

6. Подготовить тест-кейсы для `PageHeader`:
   - показывает заголовок;
   - рендерит actions справа;
   - корректно работает без actions;
   - сохраняет доступную структуру заголовка.

7. Подготовить тест-кейсы для `RefreshButton`:
   - вызывает обработчик клика;
   - показывает loading-состояние;
   - блокируется во время загрузки;
   - остается доступной для screen reader.

8. Подготовить тест-кейсы для `EmptyState`:
   - показывает иконку, заголовок и описание;
   - корректно работает без описания;
   - корректно работает без action;
   - может использоваться повторно на разных вкладках.

9. Подготовить тест-кейсы для страницы "Главная":
   - показывает блок "Истекающие абонементы";
   - показывает empty state, если данных нет;
   - показывает список абонементов, если данные есть;
   - показывает loading-состояние при обновлении;
   - показывает ошибку при неудачной загрузке.

## 3. Ревью подготовленного плана юнит-тестов до их написания

1. Проверить, что тесты покрывают пользовательское поведение, а не детали реализации.

2. Убедиться, что тесты не завязаны на:
   - случайные CSS-классы;
   - внутреннюю структуру DOM;
   - порядок вспомогательных wrapper-элементов;
   - временные детали реализации.

3. Проверить, что общие компоненты покрываются один раз:
   - навигация тестируется в `NavigationTabs`;
   - карточка страницы тестируется в `PageCard`;
   - empty state тестируется в `EmptyState`;
   - вкладки не дублируют одинаковые проверки.

4. Уточнить спорные сценарии до реализации:
   - как определяется активная вкладка;
   - как выглядит loading для кнопки "Обновить";
   - что показывать при ошибке загрузки;
   - должен ли empty state иметь action.

5. После ревью зафиксировать финальный список тест-кейсов и только затем переходить к написанию тестов.

## 11. Состояния данных

1. Empty state:
   - нет истекающих абонементов.

2. Data state:
   - есть истекающие абонементы;
   - отображается список или таблица;
   - структура соответствует общему стилю приложения.

3. Loading state:
   - при первичной загрузке;
   - при обновлении через кнопку "Обновить".

4. Error state:
   - при неудачной загрузке;
   - с возможностью повторить запрос.

   - темно-зеленый.

2. Акцент:
   - зеленый градиент для активной вкладки.

3. Радиусы:
   - `20-28px` для больших блоков;
   - `999px` для nav-кнопок и pill-элементов.

4. Тени:
   - мягкие;
   - без резкого контраста;
   - единые через дизайн-токены.

5. Общий стиль:
   - чистый CRM-интерфейс;
   - много воздуха;
   - аккуратная плотность;
   - без декоративных элементов, которые не помогают работе.
