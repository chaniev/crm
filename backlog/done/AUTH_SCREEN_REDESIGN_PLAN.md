# План доработки стартового экрана авторизации

Дата: 2026-04-23

Основание:
- mobile-анализ стартового экрана по скриншотам `IMG_1928.PNG` и `IMG_1929.PNG`;
- implement-ready spec для auth-экрана;
- текущая реализация в [frontend/src/App.tsx](../../frontend/src/App.tsx) и [frontend/src/App.css](../../frontend/src/App.css).

## 1. Цель

Превратить стартовый экран авторизации в action-first интерфейс:

- на mobile пользователь должен сразу видеть форму входа и кнопку действия;
- на desktop форма должна быть главным интерактивным центром, а левая колонка должна объяснять ценность продукта, а не внутреннюю реализацию auth-flow;
- сценарии `обычный вход` и `первичная настройка` должны быть разведены по приоритету.

## 2. Что не входит в объем

- полная замена визуального языка продукта;
- смена дизайн-стека `Mantine + Onest`;
- переработка backend-контракта за пределами небольшого флага bootstrap-режима, если он понадобится;
- новые route-сценарии вне login / forced password / bootstrap error.

## 3. Ключевые решения

- убрать с пользовательского auth-экрана внутренние термины: `Stage 4`, `MVP`, `CSRF`, `HttpOnly cookie`, `dev proxy`, `role-aware shell`;
- на mobile форма должна идти первой и в DOM, и визуально;
- на desktop можно сохранить split-layout, но с новым смысловым балансом: product context слева, auth action справа;
- блок с bootstrap-данными должен стать вторичным disclosure-блоком и не мешать обычному входу;
- те же принципы должны применяться к login-state, forced-password-state и backend-error-state.

## 4. Пошаговый план

### Шаг 1. Упростить контент и развести сценарии

Что делаем:
- заменить технический storytelling на короткий пользовательский copy;
- убрать из auth-экрана `Stage 4`, `Gym CRM MVP`, техописание про cookie/proxy и длинные bullets;
- разделить copy для:
  - обычного входа;
  - первичной настройки;
  - обязательной смены пароля;
  - ошибки загрузки backend.

Файлы:
- [frontend/src/App.tsx](../../frontend/src/App.tsx)

Готово, когда:
- заголовки и подписи помогают понять следующее действие;
- на экране входа нет текста, который нельзя использовать для выполнения входа;
- bootstrap-подсказка больше не выглядит частью стандартного login-flow.

### Шаг 2. Перестроить `StageFrame` под mobile-first приоритет

Что делаем:
- поменять DOM-порядок в pre-auth frame так, чтобы основной action-card рендерился раньше narrative-блока;
- добавить в layout явные области для desktop и mobile, а не просто stacking существующих колонок;
- на mobile вывести компактный `auth-header`, а narrative-content увести ниже формы;
- на desktop оставить split-layout через `grid-template-areas`.

Файлы:
- [frontend/src/App.tsx](../../frontend/src/App.tsx)
- [frontend/src/App.css](../../frontend/src/App.css)

Готово, когда:
- на mobile первым смысловым блоком и первым фокусируемым сценарием является форма;
- screen reader получает форму раньше narrative-content;
- desktop-композиция не ломается после смены DOM-порядка.

### Шаг 3. Пересобрать структуру `LoginScreen`

Что делаем:
- зафиксировать порядок: `error -> login -> password -> CTA -> setup disclosure`;
- убрать любой вторичный контент между полем пароля и кнопкой `Войти`;
- заменить `hint-card` на disclosure / accordion / spoiler с коротким setup-content;
- заменить placeholder `headcoach` на нейтральный placeholder или убрать его совсем;
- сохранить текущую валидацию, loading-state и error-handling.

Файлы:
- [frontend/src/App.tsx](../../frontend/src/App.tsx)

Готово, когда:
- CTA находится сразу под полем пароля;
- bootstrap-подсказка не мешает обычному входу;
- layout формы не прыгает при loading и error.

### Шаг 4. Ввести отдельный mobile-режим для pre-auth экранов

Что делаем:
- добавить breakpoint-правила для auth-layout, story-panel и stage-card;
- уменьшить на mobile размеры заголовков, padding, gap и радиусы;
- убедиться, что при ширине около `390px` первый экран показывает форму, а не длинный hero;
- скрыть или свернуть secondary content на mobile по умолчанию.

Файлы:
- [frontend/src/App.css](../../frontend/src/App.css)

Готово, когда:
- при размере экрана `390x844` пользователь видит экран входа без обязательного длинного скролла через story-panel;
- кнопка `Войти` находится близко к полю пароля;
- клавиатура на телефоне не делает CTA труднодоступной.

### Шаг 5. Доработать desktop-экран без потери текущего визуального языка

Что делаем:
- сохранить двухколоночную структуру, но уменьшить доминирование story-panel;
- сместить смысловой акцент на auth-card как главный action-блок;
- заменить технические bullets на 3 коротких product-value пункта;
- сократить пустое пространство в левой колонке и собрать её плотнее;
- при необходимости уменьшить контраст/масштаб служебных badge-элементов.

Рекомендации по desktop:
- заголовок narrative-блока: `Gym CRM для спортивного клуба` или `Управление клубом в одной системе`;
- короткое описание: `Работа с клиентами, расписанием и командой в одном интерфейсе.`;
- value points:
  - `Клиенты и абонементы`
  - `Расписание и посещаемость`
  - `Команда и роли доступа`
- форма должна визуально читаться как главный рабочий блок, а не как вторичная колонка к hero.

Файлы:
- [frontend/src/App.tsx](../../frontend/src/App.tsx)
- [frontend/src/App.css](../../frontend/src/App.css)

Готово, когда:
- desktop-экран выглядит как вход в продукт, а не как презентация внутренней архитектуры;
- правая карточка воспринимается как основная цель страницы;
- левая колонка даёт продуктовый контекст и не перегружает пользователя.

### Шаг 6. Применить тот же подход к `forced password` и `backend error`

Что делаем:
- привести все pre-auth состояния к общей модели: `action-first`, короткий copy, компактный header;
- убрать длинный технический narrative из экрана ошибки backend;
- убедиться, что forced password screen на mobile не повторяет текущую проблему большого hero перед формой.

Файлы:
- [frontend/src/App.tsx](../../frontend/src/App.tsx)
- [frontend/src/App.css](../../frontend/src/App.css)

Готово, когда:
- login, forced password и backend error выглядят как семейство согласованных pre-auth экранов;
- ни один из этих экранов не заставляет пользователя сначала читать длинный story-block.

### Шаг 7. Решить вопрос bootstrap-режима

Что делаем:
- проверить, есть ли в `session`-контракте явный признак bootstrap-режима;
- целевое решение: добавить backend-driven флаг и показывать setup-disclosure только при реальном bootstrap-сценарии;
- если backend-контракт не меняется в рамках этой задачи, использовать временный fallback: disclosure доступен всегда, но свернут, визуально вторичен и не выглядит стандартным способом входа.

Текущее состояние:
- в текущем frontend-контракте явного bootstrap-флага нет.

Рекомендуемый контракт:

```ts
export type SessionResponse = {
  isAuthenticated: boolean
  csrfToken: string
  user: AuthenticatedUser | null
  bootstrapMode: boolean
}
```

Альтернатива, если нужны будущие расширения:

```ts
setupMode: 'normal' | 'bootstrap'
```

Файлы:
- [frontend/src/lib/api.ts](../../frontend/src/lib/api.ts) при изменении контракта;
- [frontend/src/App.tsx](../../frontend/src/App.tsx);
- backend только если будет принято решение добавлять новый флаг.

Готово, когда:
- bootstrap-подсказка не воспринимается как стандартный способ входа;
- поведение подсказки предсказуемо и одинаково для всех пользователей.

### Шаг 8. Проверка, визуальная приемка и регрессия

Что делаем:
- вручную проверить pre-auth flow на mobile и desktop;
- отдельно проверить переходы: чистый login, ошибка логина, forced password, backend unavailable;
- проверить tab-order и focus behavior;
- прогнать frontend-проверки.

Минимальные команды:

```bash
cd frontend
npm run lint
npm run build
```

Желательно дополнительно:

```bash
cd frontend
npm run test:e2e -- e2e/auth.spec.ts
npm run test:e2e -- e2e/responsive-main-screens.spec.ts
```

Готово, когда:
- mobile-вход не требует лишнего скролла через narrative-блок;
- desktop-сцена выглядит собранной и продуктовой;
- проверки `lint` и `build` проходят;
- auth-flow не регрессировал.

## 5. Порядок внедрения

Рекомендуемый порядок работы:

1. Обновить copy и структуру `LoginScreen`.
2. Перестроить `StageFrame` и DOM-порядок.
3. Добавить mobile CSS и отдельные breakpoints.
4. Отполировать desktop narrative-блок.
5. Применить шаблон к `forced password` и `backend error`.
6. Добавить backend-флаг bootstrap-режима или временно оставить свернутый frontend-only disclosure.
7. Провести визуальную приемку и проверки.

## 6. Критерии приемки

- На mobile auth-форма видна сразу и не спрятана под большим hero.
- Между полем пароля и кнопкой входа нет вторичных блоков.
- На desktop левая колонка объясняет продуктовую ценность, а не устройство backend/frontend.
- Bootstrap-помощь не доминирует над обычным входом.
- Все pre-auth состояния используют единый, более компактный и понятный UX-подход.

## 7. Решения по открытым вопросам

### 7.1. Badge окружения на auth-экране

Решение:
- не показывать постоянный badge окружения на production/auth-экране;
- убрать из pre-auth narrative технические badge вроде `Stage 4`, `MVP`, `CSRF`, `dev proxy`;
- если индикатор среды нужен для разработки или staging, показывать маленький служебный badge только в non-prod окружениях: `Локальная среда` / `Тестовая среда`.

Почему:
- auth-экран должен восприниматься как вход в продукт, а не как dev-стенд;
- environment badge полезен только для предотвращения путаницы между production, staging и local;
- постоянные технические badge конкурируют с основным действием — входом.

### 7.2. Bootstrap-disclosure

Решение:
- целевой вариант — показывать bootstrap-disclosure только по backend-driven флагу из `SessionResponse`;
- рекомендуемый флаг: `bootstrapMode: boolean`;
- если нужен более расширяемый контракт, использовать `setupMode: 'normal' | 'bootstrap'`;
- временный frontend-only fallback допустим: disclosure всегда доступен, но свернут и оформлен как вторичная помощь для первого запуска.

Почему:
- frontend сейчас не знает, актуален ли bootstrap-сценарий для конкретного окружения;
- если подсказку с bootstrap-логином показывать всем, пользователь может воспринять `headcoach / 12345678` как штатный способ входа;
- backend-driven флаг делает поведение предсказуемым и одинаковым для всех пользователей.

### 7.3. Narrative-блок на mobile

Решение:
- на mobile не рендерить полный narrative/story-блок;
- оставить только компактный `auth-header`: название продукта и одну короткую строку контекста;
- product-value блок с 2-3 пунктами оставить только для desktop/tablet, где он не мешает форме.

Почему:
- на mobile приоритет должен быть `заголовок -> форма -> CTA -> ошибка/служебная помощь`;
- даже ниже формы большой narrative-блок остается лишним весом и увеличивает длину pre-auth экрана;
- desktop может объяснять продуктовую ценность, но mobile должен быть максимально action-first.
