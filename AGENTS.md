# AGENTS.md

## Назначение

Этот файл задает рабочие правила для выполнения `IMPLEMENTATION_PLAN.md` и маршрутизацию по специализированным агентам.

Использовать вместе с документами в таком порядке приоритетов:

1. `MVP-ТЗ.md` как источник истины по бизнес-требованиям и критериям приемки.
2. `IMPLEMENTATION_PLAN.md` как источник истины по этапам, архитектурным решениям и порядку реализации.
3. `AGENTS.md` как инструкция по способу выполнения работ и выбору агента под задачу.

## Как выполнять план

- На каждом шаге сначала читать соответствующий этап в `IMPLEMENTATION_PLAN.md` и связанные ограничения в `MVP-ТЗ.md`.
- Делать минимальное безопасное изменение, которое продвигает один рабочий пользовательский сценарий.
- Сначала фиксировать границу задачи: entry point, затронутые данные, права доступа, внешние зависимости.
- Не переносить проверку прав на frontend. Источником истины по доступам остается backend.
- Для каждой заметной правки проверять успешный сценарий, один сценарий ошибки и одну интеграционную границу.
- Если меняются контракты, структура проекта или важные технические решения, обновлять документацию в репозитории.
- Пока в репозитории нет исходного кода, приоритетным считать `Этап 0. Подготовка проекта`.

## Правила использования агентов

- Специализированных агентов использовать только тогда, когда делегирование явно запрошено пользователем или уже согласовано в рамках работы.
- Не отдавать агенту блокирующую задачу, если следующий шаг прямо зависит от ее результата и быстрее сделать ее локально.
- Делегировать только ограниченные по области задачи с понятной зоной ответственности.
- Если задача затрагивает несколько слоев, делить работу по границам: backend, frontend, тесты, контейнеризация, UI-решения.
- Если нужен только выбор интерфейсного решения без кодинга, сначала использовать `ui-designer`, затем реализовывать решение локально или через профильного кодового агента.

## Выбор агента по типу задачи

| Тип задачи | Основной агент | Когда использовать |
|---|---|---|
| Доменные сущности, сервисы, async/await, DTO, транзакции, side effects, прикладная логика `.NET` | `csharp-developer` | Когда задача в основном про бизнес-логику, сервисный слой, persistence behavior или контрактную безопасность |
| `ASP.NET Core` API, middleware, auth cookie, CSRF, DI, конфигурация, hosting, pipeline запросов | `dotnet-core-expert` | Когда задача в основном про HTTP-поведение, авторизацию, политики доступа, конфигурацию окружений или API boundary |
| Компоненты `React`, формы, состояние, эффекты, маршруты, доступность, поведение экранов | `react-specialist` | Когда задача в основном про клиентский интерфейс, state flow и user interaction |
| Решения по UI/UX, иерархии, состояниям `loading/empty/error`, фокусу, keyboard navigation | `ui-designer` | Когда нужен implement-ready дизайн до начала верстки или перед переработкой экрана |
| Автотесты, regression coverage, test harness, фикстуры, стабилизация CI тестов | `test-automator` | Когда нужно покрыть риск тестами или исправить инфраструктуру тестирования |
| `Dockerfile`, `docker-compose`, контейнерный runtime, healthcheck, image hardening, startup/shutdown | `docker-expert` | Когда задача про сборку, контейнеризацию, локальный запуск и эксплуатационное поведение |

## Быстрые правила выбора между `csharp-developer` и `dotnet-core-expert`

- Выбирать `csharp-developer`, если задача в первую очередь про доменную модель, прикладной сервис, вычисления, версионирование сущностей, запись в БД или безопасное изменение контракта между слоями.
- Выбирать `dotnet-core-expert`, если задача в первую очередь про API, middleware, cookie auth, CSRF, пайплайн запросов, конфигурацию приложения, DI и политики доступа.
- Если задача реально делится на две независимые части, backend business logic передавать `csharp-developer`, а HTTP/API boundary передавать `dotnet-core-expert`.

## Что особенно важно для этого проекта

- `MVP-ТЗ.md` прямо запрещает переносить контроль доступа только на frontend.
- Для тренера нужно строго ограничивать поля клиента и доступ только к назначенным группам.
- `ClientMembership` требует аккуратного версионирования и проверки правила "не более одной текущей записи".
- Проблемы с абонементом не должны блокировать отметку посещения, а только показывать предупреждение.
- `AuditLog` должен оставаться append-only и не хранить пароль или хеш пароля.

## Ожидаемый результат каждого шага

- Изменение соответствует текущему этапу плана и не тянет лишний рефакторинг.
- Есть понятный список затронутых файлов и проверок.
- Изменение двигает проект к работающему вертикальному инкременту.
- После завершения шага остаются зафиксированы риски и следующие точечные действия.
